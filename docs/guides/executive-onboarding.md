## 0. 30秒サマリ

shiki-platform（シキ）は **企業向けのAIプラットフォーム**です。一言でいうと:

> **「秘密情報を守ったまま AI を社内に入れ、AIに仕事を任せ、自社専用の業務アプリが現場でどんどん増えていく」基盤。**

解決する課題は大きく3つ:

1. **権限を守るAI検索** — 自分が見てよい社内文書だけを対象にAIが調べて答える。機密情報の混入を防ぐ。
2. **自律エージェント** — AI が安全な隔離環境の中で、ファイル作成・編集・作業を自分で片付ける。
3. **業務アプリの自動増殖** — 申請・台帳・案件管理といった社内ツールを、AI入りで簡単に作れる（**kintone の置き換え**を狙う）。

主なターゲットは、**大企業・官公庁・金融**など、セキュリティと監査が特に厳しい組織。
社内の閉じた環境（オンプレミス）でも全機能が動くことを最優先にしています。

## 1. 何を解決する製品か

### 1.1 普通の「AI導入」との違い

世の中のAIツール（一般的なチャットAIなど）を社内に入れると、こんな不安が出ます:

- 「**見てはいけない情報**（人事給与・経営情報・他部門の機密）まで、AIが回答に混ぜてしまわないか？」
- 「業務を任せたAIが、**勝手に外部へ情報を持ち出さないか？**」
- 「誰が何を見て、AIが何を根拠に答えたのか、**後から監査できるのか？**」

shiki は、この「**セキュリティ**」と「**AIの便利さ**」を**両立**させることを設計の出発点にしています。
あとから機能をセキュアにするのではなく、**最初から安全が構造に組み込まれている**のが特徴です。

### 1.2 ターゲットと提供形態

| 項目 | 内容 |
| --- | --- |
| 主な顧客 | 大企業・官公庁・金融など、セキュリティ／監査要件が厳しい組織 |
| 優先する提供形態 | **オンプレミス**（顧客の自社環境内で全機能が完結。外部インターネットに繋がない運用にも対応） |
| 将来のオプション | クラウド版（顧客ごとに隔離された環境で提供） |

> **覚え方**: 「外部のAIサービスに社内文書を預けたくない」という堅い組織が主戦場。
> だから**外に出さず、自前の環境で完結できる**ことが何より重要。

## 2. 独自用語・考え方の辞典（本資料の中心）

社内ではここに並ぶ言葉が日常的に飛び交います。**たとえ話＋一言の定義**で押さえてください。

### 2.1 コア概念（最重要）

| 用語 | 一言でいうと | たとえ話 |
| --- | --- | --- |
| **権限考慮RAG**（ごんげんこうりょ） | あなたが**見てよい文書だけ**を対象にAIが調べて答える検索機能 | 図書館で、あなたが入れる書架の本だけを司書が探してくれる。立入禁止の棚の本は最初から結果に出てこない |
| **二段authz / 二重ゲート**（にだんオーソ） | 「**アプリに許した権限**」と「**使う本人の権限**」の**両方**を満たしたものだけ通す仕組み | 入館証（アプリ）と本人確認（個人）の**両方**を受付で確認。どちらか一方では中に入れない |
| **単一チョークポイント** | 重要な処理（保存・権限判定・AI呼び出し）は必ず**1つの窓口**を通す設計 | 会社の出入口を1つに絞り、そこで全員チェック。裏口がないので「チェック漏れ」が起きない |
| **サンドボックス隔離** | 信頼しないプログラムを**頑丈な箱の中**だけで動かし、外に悪さできないようにする | 危険物を防爆室の中で扱う。中で何が起きても、外（社内システムや外部）には漏れない |
| **ミニアプリ** | shiki の上で作る**社内の業務アプリ**（申請・台帳・案件管理など） | kintone で作る社内アプリの shiki 版。さらに**AIを組み込める**のが強み |
| **自律エージェント** | 指示を受けて、AIが**自分で手順を考えて作業を片付ける**仕組み | 優秀なアシスタントに「これお願い」と頼むと、調べて・作って・直してまで一通りやってくれる |
| **fable5 委譲**（フェーブル） | 開発のうち、**正しさが極めて重要な難所**を高性能AI（fable5）に任せ、**全体設計や体験の判断は人が握る**分担 | 設計図と方針は人が決め、緻密で危険な部品加工は専用の高精度マシンに任せる |

### 2.2 「誰が何を見てよいか」まわり（認証と認可）

shiki の設計はほぼ全部「**誰が・何に・どこまでアクセスしてよいか**」の話です。土台になる言葉:

| 用語 | 一言でいうと | 補足 |
| --- | --- | --- |
| **認証**（にんしょう / AuthN） | 「**あなたは誰か**」を確かめること（ログイン） | これは「本人確認」 |
| **認可**（にんか / AuthZ） | 「**あなたはこれをしてよいか**」を判定すること（アクセス制御） | これは「権限チェック」。**ログインできた＝何でもできる、ではない** |
| **ReBAC**（リーバック） | 「**Aさんはこのフォルダの編集者**」のような**関係**を積み上げて権限を決める方式 | 「課長だから全部見られる」式（役職ベース）より柔軟で、部署階層や個別共有に強い |
| **Keycloak**（キークローク） | 認証を担当する、定評あるオープンソース基盤。**自作しない** | 顧客の既存ID（社内アカウント基盤）とも連携できる |
| **スコープ** | アプリに渡す**権限の上限**（「このアプリは閲覧まで」など） | 上限を超える操作は、たとえ本人にその権限があっても、そのアプリ経由ではできない |
| **confused deputy**（混乱した代理人） | **強い権限を持つ代理人が、権限の弱い人にだまされて**代わりに悪さをしてしまう問題 | これを防ぐのが上の**二重ゲート**。だから「本人が見られないもの」は何をどう頼んでも届かない |

### 2.3 データ・基盤まわり

| 用語 | 一言でいうと | 補足 |
| --- | --- | --- |
| **permission-aware**（パーミッションアウェア） | 「**権限を考慮した**」の意。検索やデータ取得が権限チェック込みで動くこと | shiki の機能名に頻出する形容詞 |
| **構造化データ** | **表形式のデータ**（申請一覧・顧客台帳など）。業務アプリの中核 | 行ごとに「この行を見てよい人」を厳密に制御できる |
| **モジュラモノリス** | 中身は機能ごとに整理されているが、**1つにまとまった**システム構成 | 部品を増やしすぎず**配布・運用が簡単**。必要になれば後から分割もできる |
| **オンプレ／エアギャップ** | 顧客の**自社環境内**で完結／**外部インターネットに繋がない**運用 | 機密性の高い組織向け。shiki は外部API無しでも全機能が成立するよう作る |

## 3. なぜこの設計なのか（経営視点）

shiki の独特な設計は、すべて「**事故を、運用の注意ではなく、構造で防ぐ**」という思想から来ています。

| 守りたいこと | よくある作り方（従来型） | shiki の考え方 | 経営上の効果 |
| --- | --- | --- | --- |
| **権限漏れを防ぐ** | 開発者が機能ごとに手作業で権限チェック → 抜け漏れが起きやすい | 保存・権限判定・AI呼び出しを**1つの窓口に集約**し、そこで必ずチェック（単一チョークポイント） | 「権限判定の漏れ」を**構造的に起こりにくく**し、監査での指摘リスクを下げる |
| **秘密情報をAI回答に混ぜない** | アプリの権限が広いと、本人が見られない情報まで取れてしまう | **アプリの権限×本人の権限**の両方を満たしたものだけ通す（二重ゲート） | 「他部門の機密が回答に混ざる」事故を防ぐ |
| **信頼しないアプリでも安全に使う** | 怪しいコードを動かすと社内システムに影響しうる | **隔離した箱の中**でだけ動かし、外部通信は原則遮断 | 第三者製の業務アプリも、中身を精査しきらずに安全に動かせる |
| **すべて監査できる** | ログが分散・不完全で、後から追えない | 操作・権限判定・AIが参照した文書まで**一元的に記録** | 不正や過失が起きても「どういう経路でこうなったか」を追跡できる |

> **覚え方**: shiki の安全性は「**みんなが気をつける**」ではなく「**気をつけなくても漏れない作り**」。
> ここが営業・提案での強い差別化ポイントになります。

## 4. 全体像（ざっくり）

技術的な詳細は不要ですが、提案・議論のときに「どんな部品でできているか」を大づかみできると便利です。
上から下へ、こんな流れで動きます:

1. **利用者** が画面を操作する
2. **画面（フロント）** … ブラウザで使うUI。AIが画面そのものを組み立てることもある
3. **7つの核心機能** … shiki の本体。次の7つがここに含まれる:
   - チャット（AIとの会話）
   - 自律エージェント（AIが自分で作業）
   - ファイル・ストレージ（社内文書の保管・共有）
   - 権限を守るAI検索（権限考慮RAG）
   - 認証・認可（本人確認と権限判定）
   - 構造化データ（表形式の業務データ）
   - ミニアプリ基盤（業務アプリを増やす土台）
4. **サンドボックス** … 信頼しないコードを隔離して動かす特別な区画
5. **裏方の依存部品** … 文書の保管庫、検索エンジン、権限エンジン、認証基盤など

> いずれの機能も、データに触れるときは必ず**「本人として」「窓口を通して」**動きます。
> これが第3章で説明した安全性の正体です。
