## 0. なぜ shiki は規制対象になりやすいか（機能 → 法令の対応表）

| shiki の機能 | 法的トリガ | 主な論点 |
|---|---|---|
| チャットスレッド共有・ファイル/フォルダ共有（FR-1, FR-2, FR-4） | **電気通信事業法** | ユーザー間の通信を「媒介」→ 届出が必要な可能性大（§1.1） |
| 顧客文書を預かるストレージ＋RAG（FR-2, FR-3） | **個人情報保護法** | 委託・安全管理措置・顧客文書中の個人情報（§2.1） |
| 外部 LLM API（Anthropic / Gemini / Azure）への送信（design §4.5） | **個人情報保護法 越境移転** | 外国第三者提供 or 委託の判断・本人同意/基準適合体制（§2.2） |
| マルチテナント課金・サブスク（FR-1.1 請求） | **特定商取引法**・消費者契約法 | 通信販売の表示義務（BtoB は適用除外だが表記推奨）（§1.2, §3.2） |
| 生成 AI チャット・自律エージェント・generative UI（FR-4〜6, 11） | **AI 推進法 / AI 事業者ガイドライン** | 努力義務・透明性・ガバナンス体制（§4） |
| RAG が顧客文書/外部文書を引用生成（FR-3） | **著作権法**（30条の4 / 47条の5） | 学習・索引・出力段階の権利処理（§4.3） |
| エンタープライズ調達（金融・製造・官公庁） | 任意だが事実上必須 | **ISMS / SOC 2 / ISMAP / P マーク**（§2.3） |
| サンドボックスで任意コード実行（FR-5） | 刑法・不正アクセス禁止法（運用） | 悪用防止・ログ・AUP（§3.4） |
| オンプレ/エアギャップ・暗号実装（NFR-2） | 外為法（輸出管理） | 将来の海外提供時に該当判断（§5） |

## 1. 届出・許認可（A）

### 1.1 電気通信事業法 — 届出（**該当可能性：高。要・専門家確認**）

- **論点**: shiki はチャットスレッド共有（FR-4）・ストレージのファイル/フォルダ共有（FR-2）を持つ。
  これは「他人の通信を媒介する」＝**電気通信役務**に当たり得る。総務省の整理では、
  **メール・DM・ファイル共有・Web会議・グループウェア（Slack/Teams 相当）は届出対象**とされる。
  - 「ユーザーが運営者とだけやり取りする／メッセージ機能がない」なら不要だが、shiki は
    **ユーザー間でスレッド・ファイルを共有する**ため、対象になる可能性が高い。
- **必要な手続**: 該当する場合、原則 **電気通信事業の「届出」**（登録ではなく届出で足りる規模が通常）。
  - あわせて改正電気通信事業法（令和4年改正）の **「外部送信規律」（いわゆる Cookie 規律・27条の12）** と
    **「利用者情報の適正な取扱い」** への対応が必要になり得る（Web に計測タグ等を入れる場合）。
- **アクション**:
  - [ ] 行政書士／通信法務に「shiki の機能セットが届出対象か」を確認（**ローンチ前必須**）。
  - [ ] 対象なら総務省へ電気通信事業者の届出。
  - [ ] 外部送信規律の該当確認（web/ の解析タグ・広告タグの棚卸し）。
  - [ ] 対象事業者になる場合、**通信の秘密**の保護体制・電気通信事業法上の管理規程整備。
- 参考: 総務省「電気通信事業参入マニュアル［追補版］」、令和4年改正の解説（§出典）。

### 1.2 特定商取引法 — 「特定商取引法に基づく表記」（**BtoB は適用除外だが掲載推奨**）

- SaaS は「役務提供」＝**通信販売**に当たり得る。「営業として締結する」BtoB 契約は適用除外だが、
  個人事業主・一般消費者の購入可能性を踏まえ、**「特定商取引法に基づく表記」ページは用意するのが実務標準**。
- サブスク（継続課金）は、継続契約である旨・金額・契約期間・解約条件の適切な表示が求められる。
- **アクション**: [ ] `特定商取引法に基づく表記` ページ（販売事業者・所在地・連絡先・料金・支払/解約条件）を整備。

### 1.3 その他の届出・許認可（現時点では原則不要、ただし要確認）

- **資金決済法**: 自社で前払式支払手段（ポイント等）を発行しないなら不要。決済は Stripe 等の収納代行に委ねる前提なら登録不要。**社内クレジット/チャージ残高を作る場合は要再検討**。
- **古物商・金融業・医療/医薬**等の業法: shiki 自体は非該当。ただし **顧客が規制業種で使う場合**、当該業界ガイドライン（金融庁 FISC、医療情報3省2ガイドライン等）への準拠を顧客から求められる → §2.3。
- **下請法**: 開発を外注する場合の自社の取引慣行に影響（プロダクト規制ではない）。

## 2. データ保護・セキュリティ（A/B 横断・最重要）

### 2.1 個人情報保護法（APPI）— プライバシーポリシーと安全管理措置（**必須**）

shiki は顧客の文書・チャットを預かり、顧客従業員（エンドユーザー）の個人情報も扱う。

- **立場の整理**（day-1 に固める）:
  - 顧客がアップロードした文書・チャット内の個人情報 → shiki は通常 **「委託」を受けた取扱事業者**。
    顧客（委託元）に対し、**委託先の監督（25条）に応えられる体制**（安全管理措置の説明・監査受入・再委託管理）が必要。
  - shiki が自社で取得するアカウント情報・課金情報・サポート問合せ → shiki が **管理者**としてプラポリ義務。
- **必須対応**:
  - [ ] **利用目的の特定・公表**（プライバシーポリシー）。
  - [ ] **安全管理措置**（組織的・人的・物理的・技術的）の整備と記録。設計の単一チョークポイント
    （StorageService / OpenFGA / llm-gateway）・監査ログ（FR-9, NFR-6）はこの根拠資料になる。
  - [ ] **委託先の監督**: 外部 LLM API・GCS・Qdrant 等を再委託として管理（DPA 締結）。
  - [ ] **漏えい時の報告**: 個人情報保護委員会への報告・本人通知の体制（重大事案は義務）。
  - [ ] **保有個人データの開示・訂正・利用停止請求**への対応窓口。
- 改正動向（2025〜）: 生成 AI・Cookie 等外部送信規律・課徴金/団体訴訟の導入など議論中。**継続ウォッチ**。

### 2.2 越境移転規制（**外部 LLM API を使うなら必須の設計判断**）

- **論点**: Anthropic / OpenAI / Google（米国法人）等へ個人データを送ると **「外国にある第三者への提供」**
  に当たり得る。EU・英国は十分性で例外だが、**米国宛は原則 本人同意 or 基準適合体制 が必要**。
- **「提供」か「委託」かの判断**:
  - API 利用で **学習に使われない・データ保持しない契約（DPA / ゼロ保持）** が取れるなら「委託」と整理し得るが、
    25条の監督が及ぶ仕組みが要る。整理は専門家確認必須（業界では「提供」前提で同意を取る保守運用も多い）。
- **必須対応**:
  - [ ] 利用する LLM プロバイダごとに **データ処理契約（DPA）・学習不使用・データ所在地**を確認し記録。
  - [ ] プライバシーポリシー／顧客契約に **移転先の国名・当該国の制度概要・移転先のポリシー**を明示
    （本人同意ルートを採る場合の法定記載事項）。
  - [ ] **設計反映**: llm-gateway（design §4.5）に「テナント単位で外部送信可否・送信先リージョンを制御」する
    ポリシ層を持たせる（オンプレ/エアギャップは外部送信ゼロで成立 = NFR-2 が強い差別化＆コンプラ的退避路）。
  - [ ] プロンプト/文書を外部送信する前の **PII マスキング/不送信オプション**を提供（エンタープライズ要件）。

### 2.3 セキュリティ認証（B・任意だがエンタープライズ調達で事実上必須）

| 認証 | 位置づけ | shiki での優先度 |
|---|---|---|
| **ISMS（ISO/IEC 27001:2022）** | 組織の情報セキュリティ管理。国内大手・金融・製造で事実上の前提 | **最優先**（国内エンタープライズ狙いの本命） |
| **ISO/IEC 27017** | クラウドサービス向け 27001 アドオン | ISMS 後に追加検討 |
| **SOC 2（Type I→II）** | クラウド運用状況の第三者保証。外資・海外向けに汎用 | 海外/外資顧客が視野なら取得 |
| **ISMAP** | 政府調達クラウドの登録制度。官公庁案件は登録が事実上必須 | 公共セクター展開時に検討（コスト大） |
| **プライバシーマーク** | 個人情報保護の組織認証。国内で広く要求 | 中堅顧客向けに有効 |

- コスト感: ISMS で初期 50〜300万円＋年次維持。商談の入口要件になりがちで、**投資対効果は高い**。
- **アクション**: [ ] 販路（国内エンタープライズ／官公庁／海外）を決め、**ISMS を起点**にロードマップ化。
  設計の監査ログ・認可チョークポイント・テナント隔離（NFR-8）は認証審査で有利に働くので、
  **ローンチ前から証跡（ログ・ポリシ文書）を貯める**。

## 3. 作成すべき規約・ポリシー・免責事項（C）

ローンチに向けて最低限そろえるドキュメント一式。**契約は弁護士レビュー必須**。

### 3.1 利用規約（Terms of Service）— **必須**

- BtoB SaaS なので **免責・責任制限条項は原則有効**（消費者契約法の制約を受けにくい）。盛り込むべき要点:
  - [ ] サービス内容・SLA（可用性目標）・保守/メンテナンス。
  - [ ] **責任制限**（損害賠償の上限＝直近 N ヶ月の利用料等、間接損害の排除）。
  - [ ] **データの取扱い**: 顧客データの所有権は顧客に帰属／shiki は提供目的でのみ処理。
  - [ ] **データ消失・バックアップ**の責任分界（バージョニング＝FR-2 はあるが免責は明記）。
  - [ ] **禁止事項（AUP）**: サンドボックス（FR-5）の悪用＝マルウェア・攻撃・不正アクセス・暗号資産採掘・
    大量送信等の禁止。違反時の停止権限。
  - [ ] **AI 生成物の取扱い**（§3.3 免責へ）。
  - [ ] 反社条項・準拠法/裁判管轄・解約/返金・規約変更手続。
- マルチサービス（shiki × skillex / FR-1.1）の **請求は Org 単位 1 枚・サービス別内訳** という構造を契約に反映。

### 3.2 特定商取引法に基づく表記 — **掲載推奨**（§1.2）

### 3.3 AI 利用に関する免責・注意事項（**必須・shiki 固有の肝**）

生成 AI・自律エージェント・generative UI を持つため、専用の免責/注意喚起を用意する。

- [ ] **正確性の非保証**: LLM 出力は誤り（ハルシネーション）を含み得る。重要判断は人間が検証する旨。
- [ ] **AI による生成である旨の明示**（透明性。AI 事業者ガイドライン／AI 推進法の透明性要請、海外の表示義務動向に整合）。
- [ ] **専門的助言ではない**: 法務・医療・税務・投資等の助言として依拠しない旨。
- [ ] **自律エージェントの実行リスク**: エージェントがファイル作成/編集/削除・コマンド実行する（FR-5）。
  破壊的操作の確認フロー・ユーザー責任・ロールバック（バージョニング）の範囲を明記。
- [ ] **生成物の権利・責任**: 出力の利用は顧客責任。第三者著作物との類似が生じ得る点の注意喚起（§4.3）。
- [ ] **入力データの取扱い**: 顧客が入力したデータが（設定により）外部 LLM に送信される条件・無効化方法。

### 3.4 プライバシーポリシー — **必須**（§2.1, §2.2 の記載事項を反映）

- [ ] 取得する情報・利用目的・第三者提供/委託（外部 LLM・クラウドを含む）・**越境移転（国名・制度・移転先ポリシ）**・
  安全管理措置の概要・開示等請求窓口・Cookie 等外部送信（該当時）・問い合わせ先。

### 3.5 その他用意すると強い文書（エンタープライズ調達で求められる）

- [ ] **DPA（データ処理契約）テンプレート**: 顧客＝委託元、shiki＝委託先、外部 LLM＝再委託の三層を明記。
- [ ] **セキュリティホワイトペーパー / Trust ページ**: テナント隔離・暗号化・認可モデル・監査・サブプロセッサ一覧。
- [ ] **SLA 文書**（可用性・サポート応答・クレジット）。
- [ ] **サブプロセッサ一覧**（GCP / LLM プロバイダ / Qdrant 等）と更新通知ポリシ。
- [ ] **AI 利用ポリシー / 受入可能利用（AUP）**。
- [ ] オンプレ顧客向け **OSS ライセンス順守（SBOM / ライセンス表記）**: Firecracker・gVisor・OpenFGA・Keycloak・
  Docling・vLLM 等の OSS ライセンス（Apache-2.0 / AGPL 等）を棚卸しし、再配布条件を確認（特に **AGPL 系**に注意）。

## 4. AI 固有の規制・ガイドライン（C/努力義務）

### 4.1 AI 推進法（人工知能関連技術の研究開発及び活用の推進に関する法律・2025年施行）

- **性格**: EU AI Act のような規制法ではなく **振興法。罰則なし**。事業者は **第7条の努力義務**
  （基本理念に沿った活用・国の施策への協力）のみ。
- ただし **重大な権利侵害事案では国の調査・事業者名公表**があり得るため、ガバナンス体制は実装すべき。
- 政府の「適正性指針」（2025年策定）が事実上の標準になる見込み → **準拠を前提に設計**。

### 4.2 AI 事業者ガイドライン（総務省・経産省、第1.2 版＝2026年3月）

- ソフトロー。**共通の指針10項目**: 人間中心／安全性／公平性／プライバシー保護／セキュリティ／
  透明性／説明責任／教育・リテラシー／公正競争／イノベーション。
- shiki は **「AI 提供者」**（プラットフォーム提供）かつ機能により **「AI 利用者」**。両立場の実践事項に対応:
  - [ ] **透明性**: AI 生成の明示・引用ソース表示（FR-3, FR-4 で実装済みの引用機能が効く）。
  - [ ] **説明責任 / 監査**: 監査ログ＋Langfuse トレースの trace_id 突合（FR-10, NFR-6）が直接の証跡。
  - [ ] **安全性 / セキュリティ**: サンドボックス隔離・egress 遮断・二段 authz（NFR-1）。
  - [ ] **ガバナンス体制**: AI ポリシー策定・リスク評価・インシデント対応フローの社内整備。

### 4.3 著作権法（RAG・学習・出力）

- **学習・索引（30条の4 / 47条の5）**: AI 学習目的・RAG 用ベクトル DB 作成等は権利制限の適用余地がある。
- **出力段階**: 既存著作物との **類似性・依拠性**があれば侵害になり得る。RAG が原文を「軽微利用」を超えて
  提示する場合は許諾が要る可能性。
- **アクション**:
  - [ ] RAG は **顧客が正当な権利/利用権を持つ自社文書**を対象にする前提を契約で明確化（顧客が権利保証）。
  - [ ] 出力に**引用元を明示**し、原文の大量転載を避ける設計（FR-3 のチャンク引用と整合）。
  - [ ] Web 検索ツール（FR-4・クラウド版）経由の外部コンテンツ取り込みは、出力での再利用範囲に注意。

## 5. オンプレ / エアギャップ版・海外展開の追加論点

- **外為法（輸出管理）**: 暗号機能（Keycloak/TLS 等）を含むソフトを **海外提供**する場合、該非判定が必要に
  なり得る。国内提供のみなら当面対象外だが、**海外顧客・海外拠点配布の前に判定**。
- **オンプレはコンプラ的に有利**: 外部送信ゼロ・顧客環境完結（NFR-2）のため、越境移転・委託の論点を
  顧客側に閉じられる。**規制業種（金融 FISC・医療3省2ガイドライン・公共 ISMAP）向けの強い売り**になる。
- **海外データ保護法**: 海外顧客を取る場合は **GDPR（EU）・各国法**が別途必要（本書は日本前提）。
