> ✅ **実装済み（2026-07-08・#156〜#165 / 6 本 stacked PR）**。以下の確定事項で 5.1〜5.11 を実装:
> - **Durable Workspace モデル**（human 承認済み・ライブ FUSE/永続サンドボックスは post-alpha のため不採用）:
>   ワークスペース = thread ごとの StorageService フォルダ（`thread.workspace_folder_id`・durable・版管理・監査・
>   書込→outbox→自動再索引）。**FUSE マウントの読み替え先**はこれ。file CRUD（`fs_list`/`fs_read`/`grep`/`fs_write`/
>   `fs_edit`/`fs_delete`）は StorageService 直読み/直書き（read-after-write 成立＝**PIT-5 の「ワークスペース直読み経路」**を
>   具現化・RAG 非同期索引とは経路分離）。`shell` のみ ephemeral sandbox に seed→exec→sync-back。
> - 同一 `agent-core` を `AgentProfile{Chat,Autonomous}` で切替（5.1）。計画（`plan` メタツール・5.2）・コンテキスト剪定（5.3）・
>   失敗ループ検出＋チェックポイント（5.5）・**ブロッキング承認ゲート**（`waiting_approval`＋`run_approval`＋監査・5.6）・
>   予算ガード（step/time/token/cost・5.7）・trace_id 相関＋自律イベントのライブ SSE（5.9）・UI（計画/承認/予算・5.11）。
> - **既知の未実装（後続）**: ①プロセス再起動を跨ぐチェックポイント run 再開（承認ベースの suspend は動作）
>   ②ページ再訪後の承認（run_id 再取得が要る・ライブセッション中は動作）③Playwright 自律 e2e（CI に MinIO＋
>   autonomous 対応の web-e2e 環境が要る・Rust e2e `autonomous_run_writes_workspace_file_e2e` で DoD を担保済み）
>   ④shell のパイプ/`&&`（brush 制約・ポストアルファ）。


> 📝 **方針転換（2026-07-05・#97・design §4.6）**: アルファのサンドボックスは **wasm ティア（Task 4.12）**。
> 本ファイルの「FUSE マウント」は wasm ティアでは「agentos 仮想FS→StorageService 直結」に読み替える
> （権限・監査・書込イベントの不変条件は同一。5.1/5.4 の依存 4.3/4.4 は 4.12 に読み替え）。
> 「任意コマンド」は wasm 化済みコマンド＋Node の範囲（フル環境は gVisor/FC 昇格・ポストアルファ）。
> `doc_search` とワークスペース直読みの分離（PIT-5）は変わらず適用。
>
> 目的: Phase 3 の**制約版 agent-core** を、Phase 4 のサンドボックス内で**フルツール**（shell／任意コマンド／
> ファイルCRUD）＋**長ホライズン**へ拡張する。FUSEマウントされたストレージ（`/workspace`）上で
> Claude Code / Codex 級のエージェントが自律動作し、書込はストレージイベント経由で自動再索引される。
> 同じ `agent-core` がチャットのツール実行と自律エージェントの双方を駆動する（差はツールセット／自律度／環境のみ）。
> 完了の定義(DoD): ユーザーが目標を与えると、エージェントがサンドボックス内で計画を立て、コード実行・
> ファイル編集・任意コマンドを多段で繰り返して目標を達成し、成果物がストレージに保存されて自動再索引される。
> その全過程（計画・ツール呼出・承認・コスト・進捗）が可観測で、egress遮断と承認ゲートで安全に縛られている。
> **製品の核であり、境界・安全方針・承認ポリシは慎重に設計する。**

## 詳細

### Task 5.1: `agent-core` 長ホライズン化（制約版ループの自律拡張）
- **area**: agent / **path**: `crates/agent-core`
- **依存**: 3.3, 4.3
- **仕様**:
  - Phase 3 の**制約版ループ**（計画→ツール→観測→継続→終了）を、**長ホライズン・自律**モードに拡張する。
    同一コアを設定（`AgentProfile`）で切替: チャット制約版＝短ホライズン／安全ツール、自律版＝長ホライズン／フルツール／FUSE環境。
  - エージェントは Phase 4 のサンドボックス（`sandbox-client` 経由でorchestratorに接続）内で実行され、
    ツール呼出は**ホスト↔VM の ツールRPC**（4.3）越しに行う。多段ステップ・自己駆動の継続条件を持つ。
  - **ツールセット非依存**を維持（`Tool` トレイト）。チャット版との差はプロファイルのみで、コアロジックを二重化しない。
  - **製品の核のため設計に深く関与する**（ホライズン境界・終了条件・安全方針を慎重に決定する）。
- **受け入れ条件**:
  - [ ] 1つの目標に対しエージェントが多段（複数ツール呼出）で自律的に進行できる
  - [ ] チャット制約版と自律版が同一コア＋プロファイル切替で動く
  - [ ] サンドボックス内のツールRPC越しにツールが実行される

### Task 5.2: 計画・サブタスク分解（plan→act→reflect）
- **area**: agent / **path**: `crates/agent-core`
- **依存**: 5.1
- **仕様**:
  - 目標を**計画（plan）→ 実行（act）→ 振り返り（reflect）**のサイクルで進める。計画は**サブタスクのリスト**として
    明示的に保持し、各サブタスクの状態（todo/doing/done/blocked）を更新しながら進行する。
  - 計画は**動的に改訂可能**（観測結果に応じてサブタスクを追加/削除/並べ替え）。深いネストではなく
    フラットなサブタスク列＋必要時の再計画を基本とする（過剰な階層化を避ける）。
  - 計画・サブタスク状態は構造化イベントとして外部に出し、可視化（5.9）と監査（5.10）に流す。
  - **方針（再計画の頻度・計画粒度・停滞検出）を慎重に設計する。**
- **受け入れ条件**:
  - [ ] エージェントが目標をサブタスクに分解し順に消化できる
  - [ ] 観測に応じて計画を改訂（サブタスク追加/並べ替え）できる
  - [ ] 計画とサブタスク状態の変化がイベントとして観測できる

### Task 5.3: コンテキスト管理・圧縮（要約／剪定／外部メモリ）
- **area**: agent / **path**: `crates/agent-core`
- **依存**: 5.1
- **仕様**:
  - 長ホライズンでコンテキスト長を超えないよう、会話/ツール履歴を**要約・剪定**する。古いツール結果の圧縮、
    冗長な観測の畳み込み、重要事実の保持を行う。トークン予算（5.7）と連動。
  - **外部メモリ**: 計画・確定事実・成果物の参照を agent-core 外（DB／FUSE上のワークスペースファイル）に逃がし、
    必要時に読み戻す（small-to-big 的にコンテキストへ再投入）。ファイルは StorageService 経由で永続。
  - 圧縮で**権限境界・引用の正しさを壊さない**こと（要約後も引用chunkの出所追跡が保てる、Task 2.7 と整合）。
  - **圧縮戦略・何を残すかの方針を慎重に設計する。**
- **受け入れ条件**:
  - [ ] 長時間セッションでコンテキスト長超過を起こさず継続できる
  - [ ] 圧縮後も確定事実・計画・成果物参照が失われない
  - [ ] 外部メモリへの退避と読み戻しが動作する

### Task 5.4: フルツールセット（shell/任意コマンド/ファイルCRUD on FUSE）
- **area**: agent / **path**: `crates/agent-core`, `crates/sandbox-client`, `crates/fuse`
- **依存**: 5.1, 4.4
- **仕様**:
  - 自律プロファイルのツール群を実装: ①**コード/計算実行**（任意言語・任意コマンドをサンドボックス内で実行）
    ②**ファイル作成/編集/削除**（FUSEマウントされた `/workspace` 上、裏で StorageService を叩く）
    ③**任意ツール・任意コマンドの実行**（shell）。Claude Code / Codex 級の操作セット。
  - ファイル操作は**全て FUSE 経由＝StorageService 経由**とし、権限チェック・監査・書込イベント発行を必ず通す
    （バケット/メタ直アクセス禁止、設計4.2 と整合）。
  - **egress はデフォルト遮断＋allowlist**（4.x）を継承。ネットワークアクセスは承認ゲート（5.6）対象。
  - 破壊系（削除・上書き・任意コマンド）は承認ゲート／予算ガードと連動（5.6, 5.7）。
- **受け入れ条件**:
  - [ ] エージェントがコード実行・ファイルCRUD・任意shellコマンドを実行できる
  - [ ] ファイル書込が FUSE/StorageService 経由で権限・監査・イベントを通る
  - [ ] egress 遮断が効き、allowlist 外への通信が遮断される

### Task 5.5: 失敗回復・自己修正・リトライ／チェックポイント
- **area**: agent / **path**: `crates/agent-core`
- **依存**: 5.2, 5.3
- **仕様**:
  - ツール失敗（コマンドエラー・パース失敗・タイムアウト）を観測に取り込み、**自己修正**して再試行する
    （盲目的リトライではなく、エラー内容を踏まえた計画修正）。同一失敗の**ループ検出**で停止/エスカレーション。
  - **チェックポイント**: サブタスク境界で進捗（計画・確定事実・成果物参照）を永続化し、中断/再開できる。
    回復不能な行き詰まりは安全に停止し、ユーザーへエスカレーション（5.6/5.11）する。
  - 失敗・リトライ・エスカレーションを構造化イベントとして可観測化（5.9）・監査（5.10）に流す。
  - **回復方針・ループ検出閾値・エスカレーション基準を慎重に設計する。**
- **受け入れ条件**:
  - [ ] ツール失敗後にエラーを踏まえて自己修正・再試行できる
  - [ ] 同一失敗の無限ループを検出して安全に停止する
  - [ ] チェックポイントから中断セッションを再開できる

### Task 5.6: 承認ゲート（破壊系・egress・高コスト操作の human-in-the-loop）
- **area**: agent / **path**: `crates/agent-core`, `crates/api`
- **依存**: 5.4, 3.9
- **仕様**:
  - Phase 3.9 のツール選択ポリシ（権限/破壊/高コスト系は明示許可）を自律エージェントへ拡張。
    **破壊的操作（削除・上書き）・egress（ネットワーク）・高コスト操作**は実行前に**承認ゲート**で停止し、
    ユーザーの confirm または**事前許可設定**（auto-approve ルール／対象スコープ限定）を待つ。
  - 承認待ちはエージェントを**ブロック**し、SSE 経由でUIへ承認要求を出す（5.11）。承認/却下の結果でループ継続/中断。
  - 承認の**全判定（誰が・何を・いつ許可/却下したか）を監査ログに記録**（5.10、trace_id 付き）。
- **受け入れ条件**:
  - [ ] 破壊系・egress操作が承認なしに実行されない
  - [ ] 事前許可設定で特定スコープを自動承認できる
  - [ ] 承認/却下が監査に残り、却下で当該操作が中断される

### Task 5.7: トークン／コスト／ステップ上限・予算ガード
- **area**: agent / **path**: `crates/agent-core`, `crates/llm-gateway`
- **依存**: 5.1, 3.2
- **仕様**:
  - 長ホライズンの暴走を防ぐ**予算ガード**: 1セッションあたりの**最大ステップ数・最大経過時間・トークン上限・
    コスト上限（円/USD）**を設定し、llm-gateway のトークン会計（3.2）と連動して超過時に安全停止する。
  - 上限接近時は**警告イベント**を出し、必要なら承認ゲート（5.6）で続行可否をユーザーに確認する。
  - プロファイル／skill（Phase 6・旧 prompt template）／組織ポリシ単位で上限を設定可能に（既定値＋上書き）。
- **受け入れ条件**:
  - [ ] ステップ/時間/トークン/コスト上限で安全に停止する
  - [ ] 消費トークン・コストがセッション単位で計上・参照できる
  - [ ] 上限値を設定で変更でき、超過前に警告が出る

### Task 5.8: 書込→ストレージイベント→自動再索引の配線確認
- **area**: agent / **path**: `crates/fuse`, `crates/storage`, `crates/rag`
- **依存**: 5.4, 2.9
- **仕様**:
  - エージェントが FUSE 経由で書いた/編集した/削除したファイルが、**StorageService の書込イベント → 増分再索引
    （Phase 2.9）**へ正しく流れ、数秒〜分で検索/引用可能になることを E2E で確認・配線する。
  - 削除/移動の索引整合（authz_tags 再評価含む）もエージェント書込で正しく追従すること。
  - エージェント書込は**バージョニング＋コンテンツアドレッシング**（設計4.2）の安全網に乗る（破壊的編集の復元可能性）。
- **受け入れ条件**:
  - [ ] エージェントが作成した成果物が自動で索引され検索/引用できる
  - [ ] エージェントによる削除/移動が索引に正しく反映される
  - [ ] エージェント書込が版管理され、過去版に戻せる

### Task 5.9: 進捗の可観測化（計画・ステップ・ツールのライブ可視化＋Langfuse）

- **受け入れ条件**:

  - [ ] 1自律セッションが Langfuse に1トレース（多段スパン）として出る
  - [ ] 進捗イベントが trace_id で監査・OTelと辿れる

### Task 5.10: 監査・セッションリプレイ（全ツール呼出・承認・成果物の追跡）
- **area**: agent / **path**: `crates/agent-core`, `crates/chat`, `crates/api`
- **依存**: 5.6, 3.8
- **仕様**:
  - 自律セッションの**全ツール呼出・引数・結果・承認判定・生成/編集/削除した成果物・消費コスト**を
    永続化し、後から**リプレイ/監査**できる（content blocks ＋監査ログ、trace_id 相関）。
  - 「誰が・どの権限で・どのファイルを・どのコマンドで操作し・何を承認したか」を辿れること（NFR-6 監査性）。
  - 引用・ファイル操作の権限判定を**操作時のユーザー権限**で記録（permission-aware の一貫、設計4.1/4.2 と整合）。
- **受け入れ条件**:
  - [ ] セッションの全ツール呼出と承認が時系列で再現できる
  - [ ] 生成/編集/削除した成果物が監査から追跡できる
  - [ ] 監査ログ↔Langfuse↔OTelが同一trace_idで突合できる

### Task 5.11: 自律エージェントUI（目標投入・進捗・承認・成果物）
- **area**: frontend / **path**: `web/`
- **依存**: 5.9, 5.6
- **仕様**:
  - 目標（タスク）の投入、**計画/サブタスクの進捗表示**、ステップ・ツール実行のライブ可視化、
    **承認ダイアログ**（破壊系/egress/高コストの confirm・事前許可設定）、コスト/予算の表示、
    生成された**成果物（ストレージファイル）へのリンク**を提供する。
  - 承認待ちで停止中であることが明確に分かり、却下/中断/再開（チェックポイントから）を操作できる。
  - チャットUI（3.10）と一貫した content blocks レンダリングを再利用する。
- **受け入れ条件**:
  - [ ] 目標を投入し、計画と進捗がライブで見える
  - [ ] 承認要求が出て、許可/却下でエージェントが続行/中断する
  - [ ] 生成された成果物をストレージで開ける／セッションを再開できる
