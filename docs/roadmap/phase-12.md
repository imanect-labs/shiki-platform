# Phase 12 — SaaS アルファ運用（管理・フィードバック・IaC・消去/DR）

> 目的: プライベートアルファを**顧客に配れる運用体制**を完成させる。
> 管理ダッシュボード2枚（顧客管理者／ベンダーコンソール）・フィードバック（二段同意）・ヘルプ・
> IaC（OpenTofu cell プロビジョニング）・テナント消去機構・バックアップ/DR・API レート制限（design §4.12）。
> 完了の定義(DoD): 「契約 → ベンダーコンソールから cell プロビジョニング → 顧客管理者がメンバー/モデル/予算を設定 →
> 利用 → フィードバックが Imanect に届く → 解約で消去証明が出る」が一気通貫で回る。
> サポートアクセスは break-glass のみで、サイレントアクセスの経路が存在しないことをテストで担保する。
>
> ⚠️ **着手前に [設計上の落とし穴](../design-caveats.md) の PIT-26〜28（共有プレーンの blast radius・利用量整合）・
> PIT-38（バックアップの整合スナップショット）を確認すること。**

## 詳細

### Task 12.1: モデルカタログ管理
- **area**: api / **path**: `crates/llm-gateway`, `crates/api`
- **仕様**: テナント管理者が許可モデルリスト・既定モデル・モデル別単価（課金単価表と同居）・
  国外処理バッジ・予算上限（org/ユーザー/アプリ別）を管理。ユーザーのモデル選択 UI と思考強度3段階はカタログ範囲内。
- **受け入れ条件**:
  - [ ] 許可外モデルの指定が API レベルで拒否される
  - [ ] 予算上限到達で該当スコープの呼び出しが抑制され通知される
  - [ ] 国外処理モデルにバッジが表示される

### Task 12.2: 顧客管理者ダッシュボード
- **area**: frontend / **path**: `web/`
- **仕様**: 使用量（トークン/コストを org・ユーザー・アプリ・ワークフロー別）・監査ログビューア＋エクスポート・
  組織ポリシー（フィードバック本文添付可否等）・プラン/請求（Stripe カスタマーポータル）。統一シェル＋
  マイクロフロントエンド（SK.6）に載せ、オンプレは単独動作。
- **受け入れ条件**:
  - [ ] 使用量が SAAS.3 の会計と一致して表示される
  - [ ] 監査ログを期間/種別で絞ってエクスポートできる
  - [ ] オンプレ構成で共有プレーン なしに単独動作する

### Task 12.3: 同意・委譲の棚卸し UI
- **area**: frontend / **path**: `web/`
- **仕様**: ワークフロー委譲（委譲者・スコープ・状態）・skill インストール・secret 利用許可を一覧・失効。
  fail-closed 停止中のワークフローと再同意導線。
- **受け入れ条件**:
  - [ ] 管理者が全委譲を一覧し任意に失効できる
  - [ ] 失効が次回実行前に効く（監査に残る）

### Task 12.4: ベンダーコンソール
- **area**: infra / **path**: 共有コントロールプレーン, `web/`
- **仕様**: テナントライフサイクル（プロビジョニング・停止・解約）・プラン/サブスク・テナント別機能フラグ・
  全テナント横断の**集約**使用量/SLO/ヘルス。**顧客データ本文に到達する API を持たない**（構造的不能）。
- **受け入れ条件**:
  - [ ] テナント作成〜停止〜解約がコンソールから実行できる
  - [ ] コンソールのどの API からも顧客コンテンツ本文が取得できない（テストで担保）

### Task 12.5: break-glass サポートアクセス
- **area**: auth / **path**: 共有コントロールプレーン, `crates/api`
- **仕様**: 顧客管理者の明示許可→時限付きアクセス→全操作監査→顧客画面にバナー。サイレントアクセス経路なし。
- **受け入れ条件**:
  - [ ] 許可なし・期限切れのサポートアクセスが拒否される
  - [ ] アクセス中バナーが顧客側に表示され、全操作が顧客の監査ログに残る

### Task 12.6: フィードバック機構
- **area**: obs / **path**: `crates/api`, 共有コントロールプレーン
- **仕様**: 👎＋カテゴリ（「やり方がわからなかった」含む）＋コメント＋run_id/trace_id を feedback ストアへ
  （既定は本文なし）。本文添付はユーザー明示チェック＋組織ポリシー許可の二重ゲート・添付事実を監査。
  ベンダー側で Langfuse/監査と trace_id 突合。
- **受け入れ条件**:
  - [ ] 既定送信に会話本文が含まれない（スキーマレベルで不能）
  - [ ] 組織ポリシーで本文添付を禁止すると UI からチェック自体が消える
  - [ ] trace_id でベンダー側テレメトリと突合できる

### Task 12.7: ヘルプ基盤
- **area**: frontend / **path**: `help/`, `web/`, `crates/rag`
- **仕様**: `help/`（md・リリース同バージョン）1ソース→①ヘルプセンター UI ②RAG 組み込みスコープ「shiki-help」
  （事前インデックス同梱・テナントデータと別インデックス・認可チェック不要の公開コーパス）。
  チャット doc_search が help に当たる。
- **受け入れ条件**:
  - [ ] 「◯◯のやり方」がチャットで help 由来の引用付きで返る
  - [ ] help インデックスがテナントデータと分離されている

### Task 12.8: IaC・cell プロビジョニング自動化
- **area**: infra / **path**: `deploy/tofu/`, CI
- **仕様**: GCP を OpenTofu で記述・**cell=モジュールのインスタンス化**。ベンダーコンソール→CI パイプライン→
  `tofu apply`→Keycloak realm・DNS・初期管理者招待まで自動化（SAAS.2 の admin API と結線）。冪等・dry-run。
- **受け入れ条件**:
  - [ ] 新規テナントの cell がコンソール操作1回で立ち上がり初期管理者がログインできる
  - [ ] 再実行が冪等で、失敗時に途中状態から再開できる

### Task 12.9: テナント消去機構
- **area**: infra / **path**: `crates/storage`, `deploy/`
- **仕様**: `purge_tenant` を全ストアへ拡張: DB 行・blob・Qdrant/Tantivy・FGA タプル・セッション・Langfuse・
  監査（削除証跡は保持）・**バックアップからの期限付き消滅**（保持期限経過で完全消滅する設計＋消滅証明の発行）。
- **受け入れ条件**:
  - [ ] 消去後、全ストアに該当テナントの痕跡がない（検証ジョブ）
  - [ ] 消滅証明（何を・いつ・どの範囲で）が出力できる
  - [ ] 監査の削除証跡だけが残る

### Task 12.10: バックアップ/DR
- **area**: infra / **path**: `deploy/`, `docs/guides/`
- **仕様**: テナント単位バックアップ・復元手順・RPO/RTO 宣言。Postgres・blob・Qdrant・FGA の
  **整合スナップショット**（同一時点への復元。PIT-38: バラバラ復元は FGA タプルとデータがズレて権限事故）。
  復元リハーサルを定期実行。
- **受け入れ条件**:
  - [ ] 単一テナントを指定時点へ復元でき、権限とデータが整合する
  - [ ] RPO/RTO が宣言値内であることをリハーサルで実証

### Task 12.11: API レート制限
- **area**: api / **path**: `crates/api`
- **仕様**: テナント単位のレート制限（workflow-engine のトークンバケット実装を API 面へ再利用）。
- **受け入れ条件**:
  - [ ] 超過リクエストが 429＋Retry-After で拒否され、他テナントに影響しない

### Task 12.12: 法令チェックリスト
- **area**: infra / **path**: `docs/legal/`
- **仕様**: 利用規約・プラポリ・特商法表記（弁護士マター）のチェックリストと、エンジニアリング対応
  （消去・レジデンシ・監査エクスポート・DPA 前提資料）の突合表。ISMS/SOC2 は別トラックとして起票のみ。
- **受け入れ条件**:
  - [ ] アルファ提供に必要な法務項目が担当・状態付きで一覧化される
  - [ ] DPA が参照する技術的事実（消去 SLA・レジデンシ・副処理者）がドキュメントで裏付く
