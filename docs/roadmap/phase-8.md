# Phase 8 — エンプラ硬化 ＆ クラウド対応

> 目的: 製品を**本番運用可能なエンプラ製品**へ硬化し、**同一コードベースのままクラウド版（GCP・顧客ごと隔離インスタンス）**を成立させる。
> 監査ダッシュボード・管理画面・監視整備で運用に耐える顔を備え、トレイト差し替えだけでクラウド/オンプレを吸収することを**検証**し、
> k8s/Helm による本番配布と顧客ごと隔離インスタンスのプロビジョニング自動化、受注用HWサイジング、バックアップ/DR・シークレット・アップグレード手順を整える。
> 完了の定義(DoD): 管理者が管理画面でユーザー/部署/ロール/権限を管理し、監査ダッシュボードで「誰が・どの権限で・どのチャンクを引用したか」を閲覧/検索/エクスポートでき、
> 同一マニフェストのパラメタ化で顧客ごとの隔離インスタンスを自動プロビジョニングでき、**アプリ本体を1行も変えずに**オンプレ↔クラウドのトレイト実装差し替えが成立し、その不変性がCIで保証される。

## 詳細

### Task 8.1: 監査ログ集約モデル＋検索インデックス
- **area**: obs / **path**: `crates/api`, `crates/authz`, `crates/storage`, migrations
- **依存**: 3.8, 2.7
- **仕様**:
  - 各サブシステムに散在する監査イベント（操作＝CRUD/共有、**権限判定**＝authz allow/deny、**引用chunk**＝RAG citation）を
    **統一スキーマの監査レコード**に正規化: `(id, ts, actor, org, action, resource, decision, trace_id, detail JSONB)`。
  - `trace_id` で Langfuse / OTel と突合可能（Task 3.8 の種を本実装化）。検索のため actor/action/resource/期間にインデックス、
    引用chunkは `(file_id, chunk_id, thread_id, message_id)` で逆引き可能に。**append-only・改竄検知（ハッシュチェーン）**を最初から。
- **受け入れ条件**:
  - [ ] 操作・権限判定・引用chunkが単一スキーマで横断検索できる
  - [ ] 各レコードから trace_id で Langfuse/OTel に辿れる
  - [ ] append-only で後からの改竄が検知できる

### Task 8.2: 監査ダッシュボード
- **area**: frontend / **path**: `web/`, `crates/api`
- **依存**: 8.1
- **仕様**:
  - 管理者向け監査ビュー: 期間/actor/action/resource/decision でのフィルタ検索、引用chunk閲覧（元文書/該当チャンクへリンク）、
    1回答に紐づく権限判定と引用の一覧表示。**CSV/JSONLエクスポート**（権限を持つ管理者のみ、エクスポート自体も監査）。
  - 権限境界: 監査閲覧ロールを持つ管理者のみアクセス。横断検索はあくまで管理者の認可コンテキスト内に閉じる。
- **受け入れ条件**:
  - [ ] 「誰が・どの権限で・どのチャンクを引用したか」を検索して閲覧できる
  - [ ] 結果を CSV/JSONL でエクスポートできる
  - [ ] 監査閲覧ロールのない利用者はアクセスできない

### Task 8.3: アカウント/管理画面
- **area**: auth / **path**: `web/`
- **依存**: 1.6, 1.1
- **仕様**:
  - 管理UI: ユーザー一覧/招待/無効化、部署（テナント内階層）の作成・編成、ロール/権限の割当、
    ファイル/フォルダ/スレッド共有の俯瞰と是正。OpenFGA relation と Keycloak ユーザーをUI上で一貫操作。
  - 部署=テナント内ReBAC構造の編集を**権限判定そのものを壊さないUI制約**（無効な relation を作らせない）で提供。
- **受け入れ条件**:
  - [ ] ユーザー・部署・ロール・権限を画面から管理できる
  - [ ] 部署階層の変更が権限継承に正しく反映される
  - [ ] 管理操作が監査ログ（Task 8.1）に残る

### Task 8.4: 管理API（Keycloak/OpenFGA 統合）
- **area**: auth / **path**: `crates/api`, `crates/authz`
- **依存**: 8.3
- **仕様**:
  - 管理画面の裏にあたるAPI: Keycloak Admin（ユーザー/グループ）と OpenFGA（タプル）を**整合させる単一経路**でCRUD。
    部署管理者への**委任管理**（自部署のみ操作可）を認可コンテキストで制限。
  - すべての管理操作は監査記録。skillex と共有プールのため Keycloak realm 操作は破壊系に明示確認。
- **受け入れ条件**:
  - [ ] ユーザー/部署/ロール/権限のCRUDが Keycloak と OpenFGA の双方に整合して反映される
  - [ ] 委任された部署管理者は自部署のみ操作できる
  - [ ] 全管理操作が trace_id 付きで監査に残る

### Task 8.6: クラウド版トレイト実装一式
- **area**: storage / **path**: `crates/storage`, `crates/llm-gateway`, `crates/rag`, `deploy/`
- **依存**: 3.x（差し替え対象トレイトが出揃う Phase 3 以降）
- **仕様**:
  - 既存トレイトの**クラウド実装を追加**（アプリ本体・トレイト定義は不変）:
    `ObjectStore`→**GCS**、Postgres→**Cloud SQL**（接続/IAM認証）、`LlmProvider`→**Vertex AI / 外部API**、
    監視エクスポータ→**Cloud Operations（旧 Stackdriver）**。`VectorStore`/`DocumentParser`/`EmbeddingProvider` はマネージド/外部の選択肢を整備。
  - 実装は **feature/設定で選択**（コンパイル時 or 起動時）。シークレット/エンドポイントは環境注入（Task 8.10）。
- **受け入れ条件**:
  - [ ] GCS/Cloud SQL/Vertex/Cloud Operations の各実装が単体で動作する
  - [ ] 設定切替のみでオンプレ実装↔クラウド実装が選べる
  - [ ] トレイト定義とアプリ本体コードに差分がない

### Task 8.7: アプリ本体不変の検証（CIゲート）
- **area**: infra / **path**: `.github/workflows/`, `crates/*`（テスト）, `deploy/`
- **依存**: 8.6
- **仕様**:
  - **「クラウド/オンプレ差し替えがトレイト実装だけで完結しアプリ本体不変」を機械的に保証**するCIゲート:
    ①同一の `shiki-server` バイナリを `onprem` プロファイルと `cloud` プロファイル両方で起動し、共通E2Eスイート（ログイン→ストレージ→RAGチャット→監査）を緑にする。
    ②差し替え対象は `crates/storage|llm-gateway|rag` 等の**トレイト実装モジュールに閉じている**ことを静的検査（アプリ本体crateにプロバイダ固有 import が無いことを grep/lint で検出）。
    ③トレイト定義・アプリ本体に対する変更が**プロファイル間で同一ビルド成果物**になることを確認（ハッシュ比較 or 同一バイナリ＋設定差のみ）。
- **受け入れ条件**:
  - [ ] 同一バイナリがオンプレ/クラウド両プロファイルで共通E2Eに合格する
  - [ ] プロバイダ固有コードがトレイト実装モジュール外に漏れるとCIが落ちる
  - [ ] トレイト定義/アプリ本体の差分ゼロが自動で保証される

### Task 8.8: k8s マニフェスト／Helm チャート化
- **area**: infra / **path**: `deploy/k8s`, `deploy/helm`
- **依存**: 0.x（compose 構成）
- **仕様**:
  - 既存 `docker compose` に**加えて** k8s マニフェスト/Helm チャートを整備（compose は小規模オンプレ用に維持）。
    `shiki-server` / `sandbox-orchestrator`（特権・別 PodSecurity）/ `ingestion-worker` と、ステートフル依存（Postgres/Qdrant/OpenFGA/Keycloak/MinIO/監視/Langfuse）を Helm 値で構成。
  - 特権サンドボックスのノード隔離（taint/affinity、KVMノード選択）、GPUノードへの推論スケジューリングを values 化。
- **受け入れ条件**:
  - [ ] Helm install でフルスタックが k8s 上に起動する
  - [ ] 特権オーケストレータと推論がノード隔離/GPU指定で配置される
  - [ ] compose と k8s が同一イメージ・同一設定キーで構成される

### Task 8.9: 顧客ごと隔離インスタンスのプロビジョニング自動化
- **area**: infra / **path**: `deploy/helm`, `deploy/provisioning`
- **依存**: 8.8, 8.6
- **仕様**:
  - **cell型（顧客ごと隔離インスタンス）**を**同一マニフェストのパラメタ化**で量産: 顧客ID/ドメイン/リソース枠/プロファイル（onprem|cloud）/シークレット参照を入力に、
    namespace（or プロジェクト）・DB/バケット・Keycloak realm・OpenFGA store・監視先を一括生成するプロビジョニング手順（スクリプト or GitOps）。
  - 顧客間で**マニフェストは共通・差分はパラメタのみ**（フォーク禁止）。デプロビジョニング（破棄）と再現性も担保。
- **受け入れ条件**:
  - [ ] パラメタ入力1つで新規顧客の隔離インスタンスが自動構築される
  - [ ] 顧客固有のフォークなしに（パラメタ差のみで）複数顧客を運用できる
  - [ ] 顧客インスタンスの破棄/再構築が再現可能

### Task 8.11: バックアップ／DR
- **area**: infra / **path**: `deploy/`, `deploy/dr`
- **依存**: 8.8
- **仕様**:
  - ステートフル要素のバックアップ: Postgres（PITR）、オブジェクトストア（GCS/MinIO のバージョン＋クロスサイト）、Qdrant スナップショット、
    OpenFGA タプル、Keycloak realm。**整合点（consistency point）**の取り方を定義（メタ↔実体↔ベクタ↔authz の不整合回避）。
  - **DR手順とRPO/RTO目標**、定期リストア演習のスクリプト。ベクタ等は**再構築可能な派生データ**として扱い復旧コストを下げる（`embedding_model_version` 単位の再索引）。
- **受け入れ条件**:
  - [ ] 全ステートフル要素のバックアップが定期取得される
  - [ ] 文書化された手順でRPO/RTO内に復旧できる（リストア演習で実証）
  - [ ] 再構築可能な派生データ（ベクタ等）の復旧戦略が定義されている

### Task 8.13: 受注用HWサイジング表
- **area**: infra / **path**: `docs/`, `deploy/sizing`
- **依存**: –
- **仕様**:
  - 受注時のHW構成見積テンプレ。**同時実行数 × モデルサイズ × コンテキスト長 → 必要VRAM**（生成LLM）に加え、
    埋め込み/reranker/OCR のGPU・CPU・メモリ、Postgres/Qdrant/オブジェクトのストレージ容量を概算する表/計算式。
  - KVM(Firecracker)有無、エアギャップ（全ローカル推論）/クラウド（外部API）の前提差を反映。代表的な小/中/大プロファイルのリファレンス構成を添付。
- **受け入れ条件**:
  - [ ] 同時実行×モデル×コンテキスト長から必要VRAMを見積もれる
  - [ ] 推論以外（DB/ベクタ/ストレージ）の容量も概算できる
  - [ ] 小/中/大のリファレンス構成が提示されている
