# 競合戦略: エクサウィザーズ exaBase とどう戦うか

> 対象競合: [exaBase 生成AI](https://exawizards.com/exabase/gpt/) / [同 自治体版](https://exawizards.com/exabase/gpt/gov/) / [exaBase Studio](https://exawizards.com/exabase/studio/)
> 作成: 2026-07。戦略の前提（競合の内部仕様）は公開マーケ情報ベースであり、§9 の検証が済むまで確度を割り引くこと。
> 関連正本: [requirements §1](../requirements.md) / [design §1・§4.1・§4.3](../design.md) / [executive-onboarding](../guides/executive-onboarding.md)

---

## 0. TL;DR（結論先出し）

1. **同じ土俵で戦わない。** exaBase が勝っている「横断的な全社員向け生成AIチャットの普及戦争」と「横断的な自治体市場」には正面から入らない。プレアルファが流通・実績・伴走・価格で殴り勝つのは構造的に無理。
2. **勝ち筋は3本柱**（いずれも exaBase の cloud-first・境界防御設計からは後付けしにくい構造差）:
   - **① 機微データ × 厳密権限**（二段authz / ReBAC / チャンク単位監査）
   - **② オンプレ / エアギャップ**（単一コードベースでローカル推論・全機能成立）
   - **③ 統合された業務自動化**（agent + ミニアプリ + workflow + skill が権限ネイティブに一体）
3. **主戦場 = 規制産業の機微データ部門**（金融・法務・人事・M&A・R&D/知財・医療・製薬）＋**高機密行政/防衛隣接**。ここは exaBase の「境界防御＋クラウド」では届かない。
4. **差別化は「約束」でなく「今見せられるデモ」で示す。** プレアルファである以上、ロードマップの厚さではなく、二段authz の"殺しのデモ"1本で刺す。
5. **戦略投資の判断が要る穴が2つ**: 認証（ISMS/ISMAP）の前倒し、伴走/CS モーション。§8 で human 判断を仰ぐ。

## 1. 競合の実像

### 1.1 3プロダクトの整理

| | exaBase 生成AI（GPT） | exaBase 自治体版 | exaBase Studio |
|---|---|---|---|

| 買い手 | 情シス／DX 部門 | 自治体 IT 部門 | **AI 開発チーム＋基盤運用チーム** |
| 実績 | 1,700社・15万ユーザー・「市場シェアNo.1」訴求 | 47都道府県中25県庁・200+官公庁 | （非公開） |

| 価格 | 非公開 | **月6.5万円〜・同時接続課金・ID無制限・使い放題** | 非公開 |

| RAG | 「高精度RAG」訴求。粒度は不明（§9） | 同左＋Deep Research | RAGOps テンプレート |
| 自動化 | AIエージェント標準搭載 | 標準搭載 | GUI フロー設計・LLM Chain |
| 定着支援 | **「導入から定着まで伴走」** | **7種の研修・週次更新・伴走3プラン** | インクリメンタル開発サイクル |

### 1.2 競合の強み（正直に評価する）

- **流通と実績。** 「市場シェアNo.1」「1,700社」「25県庁」は、堅い組織の稟議で効く最強のリファレンス。プレアルファには無い資産。
- **伴走・研修という GTM エンジン。** exaBase の本当の堀は製品よりも「使われるまで面倒を見る」オペレーション。"使われないAI" を潰す仕組みを商品化している。
- **認証済み。** ISO27001/27017/27701 は「セキュリティ重視組織」の入場券。我々は未取得（§8）。
- **価格の land-grab。** 自治体版の「同時接続課金・ID無制限・使い放題」は、全庁展開を安く始めさせて囲い込む設計。

### 1.3 競合の構造的な弱点（＝我々の攻め口）

- **境界防御どまり。** 訴求は ISO・IP制限・学習遮断・国内サーバ＝**組織単位の perimeter security**。「同じ組織の中で、**誰が何を読めるか**」の per-user データ境界は前面に無い。→ **§3-① が刺さる。**
- **クラウド前提。** GPT はクラウド専用、Gov も AWS東京。**真のオンプレ／エアギャップ（外部遮断で全機能）は構造的に持ちにくい。** → **§3-② が刺さる。**
- **チャットと開発基盤の分断。** 現場が使う GPT と、AI 開発チームが使う Studio が別物。**業務現場が権限ネイティブに業務アプリを増殖させる**一体基盤は無い。→ **§3-③ が刺さる。**

## 2. 彼我の非対称性マップ（どこで殴り、どこで殴らないか）

| 軸 | exaBase が勝つ | shiki が構造的に勝てる | 我々の方針 |
|---|---|---|---|
| 横断チャットの普及・ブランド | ◎ | △（プレアルファ） | **正面回避** |
| 横断的な自治体市場 | ◎（LGWAN・25県・価格） | △ | **正面回避／高機密ニッチのみ** |

| **per-user 厳密権限 RAG** | △（境界防御） | **◎（二段authz/ReBAC）** | **主戦場** |

## 3. 勝ち筋（ポジショニングの核）

### ポジショニング一文

> **「生成AIチャットは、もう入れた。次は "最も機密性の高いデータに、権限を一切崩さず AI を向け、そのデータの上で業務を自動化する" フェーズ。そこは横断チャットでは構造的に届かない。」**

shiki は "もう一つの社内 ChatGPT" ではない。**機微データとガバナンスに一切妥協できない組織のための、セキュリティ・ネイティブな AI 業務基盤**である。

### ① 機微データ × 厳密権限（最優先の刺し口）

- **主張**: 「他部門・他人の機密が、AI の回答に**構造的に**混ざらない」。二段authz（pre-filter＋post-filter）＋ ReBAC ＋ チャンク単位の引用監査（design §4.3）。
- **なぜ exaBase が追随しにくいか**: 彼らの安全は「学習させない・外に出さない・IPで囲う」= 境界。**同一組織内の per-user 可読性**を検索の心臓部に組み込むのは、cloud-first・横断展開・高速普及を優先した設計思想からは後付けが重い。ここは製品思想の分岐点であり、機能追加で埋まらない。
- **刺さる先**: 部門横断で権限が複雑／機微データを持つ組織。**金融（M&A・与信・トレーディング情報）・法務・人事・研究開発/知財・医療・製薬。** これらでは「フラットな RAG」はそのまま**情報漏洩リスク**であり、導入のブロッカーになる。
- **証明方法（殺しのデモ・§6）**: 同一の質問を権限の異なる2ユーザーが投げ、**証明可能に異なる・正しく境界された回答**が返り、各回答のチャンク引用が監査に残る。これが最強の一枚。

### ② オンプレ / エアギャップ

- **主張**: 単一コードベースで SaaS ↔ オンプレ ↔ **エアギャップ（外部遮断で全機能成立・ローカル推論）**。トレイト差し替えのみでアプリ本体不変（NFR-2/3）。
- **なぜ exaBase が追随しにくいか**: GPT はクラウド専用、Gov も AWS東京。オンプレ/エアギャップは製品ラインの作り直しに近い。
- **刺さる先**: 防衛・重要インフラ・厳格な隔離要件の金融・**高機密行政（LGWAN の外側／中央省庁の機密系）**。exaBase Gov がクラウドで取れない上澄み。
- **注意**: 「設定差し替えだけでオンプレ化」は過大主張しない（[PIT-26〜29](../design-caveats.md)）。エアギャップは"機能単位の縮退"を正直に見せる方が、堅い顧客には却って信頼される。

### ③ 統合された業務自動化（拡大フェーズの本命）

- **主張**: 業務現場が、権限ネイティブに **ミニアプリ＋ワークフロー＋自律エージェント** を増殖できる一体基盤。全て同じ ReBAC/監査/単一チョークポイントに乗る（FR-6/11/12/14）。
- **exaBase との差**: 彼らは GPT（チャット）と Studio（K8s 開発基盤・AI 開発チーム向け）が別物。**Studio と正面で殴らない** — あちらは開発者向け PaaS、こちらは**情シス/現場が K8s 無しで業務アプリを増やす**基盤。**打倒 kintone / Power Automate** として位置づける。
- **効き方**: ミニアプリが増えるほど lock-in が効き、チャット専業の incumbent には追随不能なスイッチングコストになる。**Land した後の ACV 拡大とスティッキネスの主役。**

## 4. ターゲット・セグメント

- **ビーチヘッド（最初に取る）**: 規制産業の**機微データ部門**を2〜3社、lighthouse として。「全社チャット」ではなく「人事/法務/M&A/研究データに安全に向けられる唯一の RAG」を単一ユースケースで刺す。
- **拡大**: lighthouse 内で ② オンプレ要件と ③ 業務自動化へ横展開（部門→全社）。
- **避ける（アンチターゲット）**: 「安く全社員にチャットを配りたい」層（exaBase の土俵）、横断的な自治体（Gov の堀）。ここに営業リソースを割かない。

## 5. Land → Expand → Moat のシーケンス

| 段階 | 武器 | 売り文句 | 依存する実装 |
|---|---|---|---|
| **Land** | ① permission-aware RAG ＋ ② オンプレ/エアギャップ | 「機微データに向けられる唯一の RAG」 | Phase 2–3（実装済み）・Phase 8 オンプレ硬化 |
| **Expand** | ③ agent ＋ ミニアプリ ＋ workflow | 「打倒 kintone、ただし AI・権限ネイティブ」 | Phase 5/6/9/10（大半実装済み） |
| **Moat** | 認証 ＋ リファレンス ＋ 単一コードベース | 「監査に耐える AI 基盤」 | ISMS/ISMAP（§8）・実績 |

現実チェック: ロードマップ上、① ② ③ の中核は**既に実装が進んでいる**（Phase 10 Stage A 完了・Office/エディタ実装済み）。**足りないのは製品ではなく、"証明"と"信頼資産"**（デモ・認証・リファレンス）。戦略の重心はそこに置く。

## 6. 製品で二重投資すべき「殺しのデモ」

1. **権限境界の証明デモ（最優先）**: 2ユーザー・同一質問 → 証明可能に異なる正しい回答 ＋ チャンク引用監査ビュー。exaBase のフラット RAG との差が一目で伝わる唯一の絵。
2. **エアギャップ全機能デモ**: LAN ケーブルを抜いた状態で RAG・エージェント・資料生成が動く。「外部遮断で全機能」の binary な安心。
3. **AI・権限ネイティブ ミニアプリ**: チャットから申請/台帳アプリが生まれ、行レベル権限と承認ワークフローが即座に効く（打倒 kintone）。

これらは機能ではなく**"体験としての差別化"**。CLAUDE.md の「使っていて楽しい／使いたくなる」は、ここでは"使われないAI"を潰す GTM 兵器として効く。

## 7. 正面回避すべき戦い（アンチゴール）

- ❌ 横断チャットの機能比較表 — 実績で負ける。自分から乗らない。
- ❌ 価格の使い放題競争 — land-grab は資本力勝負。差別化を価格に落とさない。
- ❌ 横断的な自治体入札 — LGWAN・25県・研修の堀。高機密ニッチだけ狙う。
- ❌ マルチモデル対応の"数"競争 — テーブルステークス。llm-gateway の抽象で十分、宣伝の主役にしない。

## 8. human の判断が必要な戦略分岐（要相談）

CLAUDE.md「判断に迷ったら必ず human に相談」に従い、**プロダクト判断を超える経営分岐**を明示する。

1. **自治体戦略の是非。** 推奨は「横断 Gov は正面回避、高機密行政/防衛隣接のオンプレのみ狙う」。全面参入は非推奨だが、事業判断として要確認。
2. **認証（ISMS→ISMAP→SOC2）の前倒し投資。** セキュリティ・ネイティブを掲げる以上、認証未取得は信頼の穴。ロードマップは「アルファ後」だが、**少なくとも ISMS のプロセス着手は前倒し**を推奨。投資・時期は経営判断。
3. **伴走/CS モーションの構築 vs セルフサーブ設計への配分。** exaBase の本当の堀は伴走。製品が優れていても定着支援ゼロでは規制産業の稟議は通らない。lighthouse 数社への lean 伴走＋"使っていて楽しい"によるセルフサーブ、の two-track を推奨するが投資配分は要判断。
4. **プレアルファの GTM タイミング。** 四半期ごとに exaBase は entrench する。差別化は「今のデモ」で示すべきで、フル機能アルファを待つと機会を逃す。lighthouse への早期 POC を推奨。

## 9. 検証すべき前提（このドキュメントの弱点＝正直に開示）

本戦略は公開マーケ情報に依存する。以下は一次情報で確度を上げること（誤っていれば §3 の刺し口が弱まる箇所）:

- exaBase の RAG が **per-user 権限**をどこまで実装しているか（我々の①の前提。もし彼らが ReBAC 級を持っていれば差は縮む）。
- exaBase GPT（法人版）の価格・**オンプレ提供の有無**（②の前提）。
- exaBase Studio の **permission-aware / authz の深度**（③の前提）。
- ISMAP 取得時期（Gov 競合の②における時間的優位の見積り）。
- 「市場シェアNo.1」の定義・出典（稟議での我々の counter-narrative 設計に影響）。
