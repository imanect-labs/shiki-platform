# shiki 要件定義書

> 本書は設計インタビュー（grill-me）で合意した内容を要件として確定したもの。
> 関連文書: [設計書](./design.md) / [ROADMAP](./roadmap.md)

## 1. プロダクト概要

shiki は **LLMチャットとストレージを核とするエンタープライズ向けAIプラットフォーム**。
権限を厳密に守った文書検索（permission-aware RAG）、サンドボックス上で自律動作するAIエージェント、
社内ツールが自然増殖する generative UI を提供する。SaaS版（マネージド）とオンプレ版の両方を、
**単一のコードベース**から提供する（SaaS を優先ターゲットとする）。

### 1.1 提供形態

| 形態 | 配布 | テナンシー | 推論 |
|------|------|-----------|------|
| **SaaS版（優先）** | Google Cloud（マネージド） | マルチテナント：**共有コントロールプレーン（Keycloak/Org/請求）＋顧客ごと隔離データプレーン（cell）**（design §4.1.1） | マネージド/外部API or ローカル |
| **オンプレ版** | k8s、小規模は docker compose | シングルテナント（1社専有） | 完全ローカル成立可（エアギャップ対応） |

- **SaaS版を最優先で開発**する。オンプレ版は「同一成果物を顧客が自社ホストする」位置づけ（4〜5本のトレイト差し替えで吸収）。
- **リリース定義（2026-07 確定・#97）**: 本書の全機能要件（FR-1〜17）を実装した完成形を**プライベートアルファ**として
  リリースし、バグ潰しと追加機能検討を進める（段階リリースで機能を削る方針は採らない）。
- **単一コードベース原則は維持**：SaaS/オンプレ差はトレイト裏とデプロイ構成で吸収し、アプリ本体は分岐させない。
- マルチテナントを最初から前提にするため、**認可コンテキストの `tenant_id` の継ぎ目を day-1 から確保**する（後付けで隔離境界を壊さない）。

## 2. 前提・基本方針

- **開発の進め方**: 機能ごとにレイヤー横断（API→サービス→ストレージ→フロント）で縦スライス実装する。
- **モダン構成**: サーバ=Rust、フロント=Next.js/TypeScript、モノレポ。
- **コアは自作**: サンドボックス制御層、RAGパイプライン、agent-core、llm-gateway、FUSEを自作する。
- **枯れた技術に乗る領域**: 隔離プリミティブ（Firecracker/gVisor）、認可エンジン（OpenFGA/SpiceDB）、
  認証（Keycloak）、文書パース（Docling）は自作しない（正しさ・運用がクリティカルなため）。
- **可搬性は4〜5本のトレイトに集約**: `ObjectStore` / `VectorStore` / `LlmProvider` / `Sandbox` /
  `DocumentParser` の実装差し替えでクラウド/オンプレを吸収。アプリ本体は無変更。

## 3. 機能要件（FR）

### FR-1 認証・認可基盤
- **認証(AuthN)**: Keycloak。OIDC/SAML/LDAP/AD フェデレーション（顧客IdP連携）＋ローカルIdPの両対応。
- **認可(AuthZ)**: ReBAC（OpenFGA または SpiceDB）。認可の真実を authz store に一元化。
- **組織階層**: ロールなどの階層をテナント内に持たせ、柔軟な権限管理。テナント=会社、ロール=テナント内ReBAC構造。
- **共有**: ストレージのファイル/フォルダ共有、チャットスレッドの共有を ReBAC relation で表現
  （閲覧/コメント/編集など）。
- **skillex統合 / マルチサービス境界**: 詳細は FR-1.1。

### FR-1.1 マルチサービス境界（shiki × skillex）
- **統一は SaaS 版限定**。オンプレ版は shiki・skillex とも**認証基盤を切り離して単独運用**
  （ローカルKeycloak＋顧客AD、外部依存ゼロ）。エアギャップ要件と矛盾させない。
- **SaaS版の二層**: マルチテナントな**共有コントロールプレーン**（Keycloak＋Org/Member＋請求＋管理ダッシュボード）の下に、
  顧客ごと隔離の**データプレーン**（shikiセル、skillexバックエンド）がぶら下がる。
- **3層の統一/分離境界**:
  1. **User（個人ID）= 統一**。1人1アカウント、共有Keycloakのアイデンティティ。
  2. **Org・メンバー招待・サービスアクセス権（＋サービス管理者の粗い区分）= 統一**（共有プレーンが保持）。
     「サービスロール付与」は `そのサービスを使えるか＋サービス管理者か` 程度の粗い粒度のみ。
  3. **サービス内の細かいロール/ReBAC・各サービスの管理/設定 = 分離**（各データプレーンが独立に保持）。
     shikiのロール階層・ReBAC（OpenFGA）は shiki 内に閉じる。skillexは自前の認可。
- **管理画面はUIのみ統一（データは分離）**: SaaS版は1つの統一「アカウント管理画面」（統一シェル＋共有ページ）に、
  各サービスの設定ページをマイクロフロントエンドで合成し、ページが分かれているだけの一体UIにする。
  ただし各ページは自分のサービスのAPI/ストアを叩き、**authz・設定データは分離のまま**。
  各サービスの管理ページは「シェル埋め込み／単独」両対応の自己完結モジュールとし、オンプレでは単独の管理画面として動く。
- **請求（統一）× 利用量（分離）**: 各サービスが利用量を自前計測（分離）し、**集約使用量のみ**共有請求プレーンへ送る。
  請求書は **Org単位で1枚・サービス別内訳**。プラン/サブスクはサービス別、束ねて1請求。
  **クォータ/上限の強制は各サービス側**、共有プレーンは金額集約のみ。
- **所有**: 共有コントロールプレーンは **shiki repo 所有の SaaS専用モジュール**。skillex は消費側
  （OIDCログイン・サービスアクセス権参照・利用量送信・DLC/LLMトークン取得）。
  shiki-server の AuthN 向き先は**設定で差し替え**（SaaS=共有issuer / オンプレ=ローカルKeycloak）。
- **契約（コントラクト）の正本は shiki repo**: skillex は**別リポジトリ**。skillex が参照する契約
  （OIDC設定、サービスアクセス権API、利用量集約イベントのスキーマ、DLC/LLMトークンの aud/scope）は
  **shiki repo の `contracts/` に正本を置き公開**する。skillex 側はこれを取り込んで実装し、齟齬を防ぐ
  （スキーマ変更は shiki repo 側のバージョン管理・後方互換ポリシで管理）。

### FR-2 ストレージ
- Google Drive 風の基本ファイル操作（アップロード/ダウンロード/フォルダ階層/移動/共有）。
- **3つの顔**を持つ: ①Drive風UI操作、②RAGインデックス対象、③AIエージェントがマウントするファイルシステム。
- **全アクセスは単一 `StorageService` 経由**（権限チェック＋監査＋イベント発行のチョークポイント）。
  バケット直アクセス禁止。
- **バージョニング＋コンテンツアドレッシング**（重複排除、破壊的編集の安全網）を最初から。
- 書込時に**イベント発行**し、RAG増分再索引をトリガ。

### FR-3 permission-aware RAG（精度の良い高速な文書検索）
- **閲覧不可ファイルは検索結果にもLLM回答にも絶対に混入しない**（厳密な権限境界）。
- **二段authzフィルタ**: ①ベクタ/全文の検索時に可読タグで pre-filter、②取得後に authz store で post-filter 検証。
- **ハイブリッド検索**: Dense（埋め込み）＋ Keyword（BM25・日本語形態素）を RRF で融合 → reranker で並べ替え。
- **日本語重視・差し替え可能な埋め込み**（デフォルト Ruri、`embedding_model_version` をベクタに刻む）。
- **高品質インジェスト**: PDF/docx/pptx/xlsx をレイアウト・表・読み順を保って構造化、スキャンPDFは日本語OCR。
- **監査**: 「どの文書のどのチャンクを引用して回答したか」をチャンク単位で記録。

### FR-4 LLMチャット
- スレッド（会話）→ メッセージ。メッセージ内容は**構造化ブロック配列**
  （text / tool_call / tool_result / citation / generative_ui / file_ref）。
- **ツール実行**: コードインタプリタ、文書検索、（クラウド版は）Web検索など。
  - **ユーザーが選択していなくてもLLMが自動判断してツールを使用**（デフォルト全提示・モデル自動選択、
    権限/破壊/コスト系のみ明示許可）。
- **引用ソースの表示**、**SSEストリーミング**（トークン・ツールイベント・生成UIを逐次配信）。
- スレッド共有（ReBAC）。初版は線形会話、ただしブランチ可能な構造。
- **バックグラウンド継続生成**: 送信後にページを離れても生成が継続する（run 抽象＋DB リプレイ＋pub/sub。design §4.4.1）。
- **モデル選択と思考強度**: テナント管理者が絞る**モデルカタログ**の範囲でユーザーがモデルを選択。
  思考強度は `low/medium/high` の3段階（プロバイダ差は llm-gateway が正規化）。
- **ツール**: web 検索（`SearchProvider`・エアギャップは無効）／code（code_interpreter）／
  **deepresearch**（専用エンジンではなく agent-core の first-party プリセット。成果物はストレージ→RAG 還流）。

### FR-5 サンドボックス＆AIエージェント
- **サンドボックス＝隔離された汎用実行環境**（プリミティブ）。その上に用途が乗る：
  - **自律エージェント**: Claude Code/Codex 級。①コード/計算の実行、②ファイルの作成/編集/削除、
    ③ツール・コマンドの実行。永続ワークスペース＋ストレージをマウント（wasm ティアは仮想FS直結）。
  - **コードインタプリタ/単発実行**: まっさら・短命・実行して破棄、ネット遮断などで強く制約。
- **バックエンドは3ティア（2026-07 改訂・design §4.6）**: **wasm（agentos フォーク・既定・アルファはこれのみ）**／
  gVisor（フル CPython・ネイティブツールチェーン）／Firecracker（VM級隔離が契約要件・GPU）。
  wasm ティアでは「任意コマンド」は wasm 化済みコマンド＋Node に縮み、code_interpreter は Pyodide
  （numpy/pandas/matplotlib）。フル環境が要るワークロードは gVisor/FC へ昇格（ポストアルファ）。
- **単一 `agent-core`** がチャットのツール実行と自律エージェントの双方を駆動（差はツールセット/自律度/環境のみ）。
- **egress デフォルト遮断＋allowlist**（機密データ持ち出し防止、エアギャップ対応）。

### FR-6 generative UI ＆ 社内ミニアプリ
- チャット内でAI生成UIコンポーネントをレンダリング。
- **段階戦略**: まず宣言的コンポーネント・カタログ（任意コード実行なし・安全）。
  これは二層モデルの **A（宣言的・in-trust）**。コードベースの **B** と業務アプリ基盤は **FR-11** で扱う。
- **生成UIにアンビエント権限を与えない**。データアクセスは必ず宣言済み・認可済みバックエンドアクション経由。
- **ミニアプリ = skill ＋ UIスペック ＋ ワークフロー**（FR-12・Phase 10 Stage A前倒しにより実装可能）の
  バージョン付きアーティファクト、ReBACでロール共有。**テーブルを含む完全な定義は FR-11 を参照**
  （UIスペック＋テーブル＋ワークフロー＋skill＋script。設計正本: [miniapp-platform.md](./miniapp-platform.md)）。

### FR-7 skill（gem/GPTs相当。旧 prompt template を統合）
> 設計正本: [miniapp-platform.md](./miniapp-platform.md) §4。呼び出し面・スキルストアは FR-14 参照
> （本FRは skill artifact の構成、FR-14 は配布・呼び出し面）。
- 構成: **① SKILL.md 相当の指示文**（name/description のフロントマター＋用途・振る舞いを書く本文。
  Claude Code の skill と同型）**② 知識スコープ**（参照を許すフォルダ/タグ＝RAG範囲限定）**③ 許可ツール**
  **④ モデル/パラメータ既定** ⑤（任意）few-shot ⑥（任意）script ⑦（任意）参照資料。
  **⑥の script は shiki script（`.shiki`・FR-13）と shell script（`.sh`）のどちらも、また両方を同時に
  含められる**（`.shiki`はscript-runtime実行、`.sh`はagent.invokeのサンドボックス内実行）。
- 知識スコープで参照範囲を絞れるが、**最終可読性は常に個人のReBACで再チェック**。
- ロール単位で共有（ReBAC）、バージョン管理。

### FR-8 資料作成・Office 統合・ドキュメント編集
- パワポ/ワード/エクセル生成: ライブラリ直接生成（xlsx=`rust_xlsxwriter`、docx/pptx=Python worker）＋ひな型穴埋め併設。
- **ブラウザ内編集・共同編集（旧 v2 を前倒し・アルファスコープ）**: **Collabora Online** を `OfficeSuite` トレイトで
  ラップし、docx/pptx/xlsx の編集・共同編集を提供（共同編集は Collabora 内蔵に委任・自作しない）。
  保存は WOPI ホスト経由で StorageService（バージョニング→RAG 再索引）。
  **AI の編集は非セッション時のファイルレベル編集のみ**（編集セッション中は「提案として保存」。design §4.8）。
- **マークダウンエディタ（新規）**: TipTap ベースの WYSIWYG（内部表現の真実は Yjs・md はシリアライズ形式）。
  **共同編集は Yjs/yrs（CRDT）**、**AI 編集も共同編集参加者として同一エンジン経由**（design §4.8.1）。
- **スプレッドシート×スクリプト**: シートのカスタム関数/マクロ = shiki script（FR-13）。ミニアプリの中核部品を兼ねる。

### FR-9 エンタープライズ機能
- アカウント管理、管理画面、**監査ログ**（操作・権限判定・引用チャンクまで）。
- **管理ダッシュボードは2枚（design §4.12）**:
  ①**顧客管理者**: メンバー/ロール・モデルカタログ/予算・使用量・監査ビューア＋エクスポート・
  **同意/委譲の一覧/棚卸し/失効**（ワークフロー委譲・skill・secret）・プラン/請求（Stripe）・組織ポリシー。
  ②**ベンダーコンソール（Imanect 運用専用・共有コントロールプレーン）**: テナントライフサイクル・プラン・
  機能フラグ・集約使用量/SLO/ヘルス（顧客データ本文には構造的に到達不能）。
- **サポートアクセスは break-glass のみ**（顧客管理者の明示許可→時限→全操作監査→バナー表示。サイレントアクセス経路は作らない）。
- **ドライブの更新者表示**: `updated_by` をノードメタ＋バージョン作者に保持し、バージョン履歴 UI（誰が・いつ・どの版）とセットで表示。
- **テナント消去・データレジデンシ**: 解約/削除要求に応える cell 完全消去＋消滅証明。東京リージョン固定・
  外部 LLM の国外処理はモデルカタログのバッジで明示（design §4.12）。

### FR-10 監視基盤
- **インフラ可観測性**: OpenTelemetry ＋ Grafana スタック（Tempo/Loki/Prometheus）。
- **LLM可視化**: Langfuse（プロンプト/補完/トークン/コスト/ツール/レイテンシ/引用チャンク）。
- **監査ログと Langfuse トレースを trace_id で突合**（「誰が・どの権限で・どのチャンクを引用した回答を得たか」）。

### FR-11 ミニアプリ／業務アプリ基盤
FR-6 の generative UI を土台に、「社内業務アプリが自然増殖する」基盤へ拡張する。

- **二層モデル**（同一の「アーティファクト＋バージョン＋ReBAC＋監査」共通枠に乗る。違いはランタイムと認可の入口だけ）:
  - **A. 宣言的ミニアプリ（=FR-6）**: LLM/利用者が作る検証済みUIスペック。in-trust・構造的に安全・宣言的バックエンド束縛のみ。
  - **B. コードベース・ミニアプリ（新規）**: shikiチーム/パワーユーザーが任意フロント（＋任意サーバ）コードで実装。
    out-of-trust として隔離実行し、shiki機能は**公開API＋スコープ付きトークン**経由でのみ呼ぶ。
    **B1＝フロントのみ（本命）／B2＝サーバ側ロジックあり（昇格オプション）**。
- **汎用PaaS/DBaaSは作らない**: アプリが要る「データ置き場／実行場所／shikiへの口」を、
  ①管理された構造化データサービス ②既存サンドボックス基盤の再利用 ③公開APIゲートウェイ、の3点で満たす。
- **セキュアな内部API呼び出し（最重要）**:
  - 内部APIは直接公開しない。**公開APIゲートウェイ（BFF）が唯一の入口**で、キュレーション済み・バージョン付き・スコープ付きの能力面だけを再公開。
  - 認可は**ユーザー委譲OAuth2（authcode＋PKCE）**。既存Keycloakを認可サーバに再利用（新規認証基盤なし）。
  - **二重ゲート**: 実効権限 = アプリ付与スコープ ∩ 呼出ユーザーのReBAC。
    アプリが広いスコープを持っても本人が閲覧不可なデータには到達不可（confused deputy 対策）。
  - ユーザー不在の自動化のみ、**所有データに限定した**狭スコープのサービスidentityを許可。全呼び出しを監査。
- **能力カタログ＆スコープ**: storage / data / rag / ai / identity / events。
  `<能力>.<操作>` ＋ リソース束縛の2軸。スコープは上限、実認可は常にOpenFGA。アプリは自分の所有リソースを持てる。
- **ミニアプリ内AI**: raw `llm.invoke` ＋ `agent.invoke`（ツール＝アプリ宣言 ∩ skill ∩ ユーザーReBAC）。
  コンテキストにアプリデータ/RAGを供給可（RAGは個人ReBAC再チェック）。コストは(ユーザー×アプリ)で計上、モデル/予算ガードレール、既存監査に統合。
- **構造化データサービス**: 既存Postgres上の管理サービス（`record` JSONB ＋ スキーマレジストリ ＋ 宣言フィールドへの式インデックス、ランタイムDDLなし）。
  フィールド型に user/dept/file/record 参照を含む。
  **行レベル認可 = テーブルReBAC（OpenFGA）＋クエリ時述語（ABAC・WHERE強制付与・集計にも適用）＋フィールドマスク＋個別共有のみスパースtuple**。
  宣言的クエリ/保存ビュー（生SQL非公開）。レコードのリビジョン履歴。
- **ワークフロー/プロセス管理（2026-07 改訂）**: 旧「軽量FSMエンジン」は廃止。状態管理は data サービスの
  **宣言的ガード**（status フィールド＋遷移認可=行述語の再利用・可視性駆動・record 書込と同一 Tx で原子的）に縮退し、
  副作用・プロセス実行は **FR-12 の workflow-engine に一本化**（遷移コミット→outbox→トリガ）。
- **ミニアプリの構成部品（改訂）**: ミニアプリ = UIスペック＋テーブル＋ワークフロー（FR-12）＋skill（FR-14）＋
  script（FR-13）を束ねたバージョン付きアーティファクト。作成手段はコードエディタ・dnd・チャット（AI 生成）の3系統
  （正本はすべて同一の検証済みスペック）。設計正本: [miniapp-platform.md](./miniapp-platform.md)。
- **ランタイム隔離**: B1=別オリジン＋CSP（connect-srcはゲートウェイ限定・ホスト無権限）／
  B2=既存サンドボックス（Firecracker/gVisor）＋egressデフォルト遮断+allowlist。
- **配布・デプロイ**: ミニアプリ=マニフェスト付きアーティファクト（要求スコープ/所有テーブル/ワークフロー/許可モデル/バンドル/エントリ）。
  内部レジストリに不変publish→管理者が同意してインストール（所有テーブル自動プロビジョン＋ReBAC付与）。
  信頼ティア（first-party署名＝既定信頼／in-house＝管理者同意／将来marketplace＝審査）。
  オンプレ/エアギャップは署名バンドルでネット不要インポート。SDK＋CLI（`shiki app init/dev/publish`）で「簡単にデプロイ」。

### FR-12 ワークフロー基盤（n8n/Power Automate 相当・自作 Durable Execution）
> 設計正本: [miniapp-platform.md](./miniapp-platform.md) §1〜2

- **workflow-engine（自作・`crates/workflow-engine`・Postgres 上・新規ステートフル依存ゼロ）を唯一の長時間実行ランタイム**とする。
  Temporal 等の外部エンジンは入れない。
- **定義の正本 = IR（JSON DAG・バージョン付き artifact）**。編集手段は **dnd エディタ**と
  **AI 編集（IR を生成する検証済みスペック方式・人間が dnd で続きを編集可能）**の2つ。
  **shiki script は IR の編集手段ではなく、ノードの実装言語**（code view / IR↔script 双方向同期はスコープ外・§6）。
  ノードの参照（ツール/スコープ/skill/secret）は保存時に codegen 語彙へ照合して拒否。
- **Durability はノード境界のみ**（決定論的リプレイ・code-first workflow は採らない）。
- 要求機能: **イベント駆動実行**（outbox 再利用）・**ステップ単位リトライ**（at-least-once・冪等推奨）・
  **長時間実行（Durable Execution）**・**スケジューリング**（cron・リーダーリース）・**並列/Fan-out・join**・
  **Concurrency 制御**（テナント/ワークフロー/ノード種の3階層）・**Rate Limit**（テナント×能力トークンバケット）・
  **Observability**（run/step 実行履歴 UI・OTel・Langfuse 突合）。
- **実行主体**: 対話トリガ=本人 ReBAC ∩ 宣言スコープ／スケジュール・イベント=ワークフロー専用サービスプリンシパル＋
  **有効化時の明示委譲**（委譲者失権で fail-closed 停止→再同意）。ノード設定（サンドボックス許可範囲等）は**縮小のみ**。
- **AI ノード2種**: `agent.invoke`（サンドボックス・ノード設定で egress/マウント/ツール/上限を細粒度指定）／
  `llm.invoke`（生 LLM 呼び出し）。
- **外部コネクタはネイティブ実装しない**: `http.request`＋シークレット参照を土台に skill（FR-14）として構成・配布。

### FR-13 shiki script（GAS 相当のスクリプト言語）
> 設計正本: [miniapp-platform.md](./miniapp-platform.md) §3

- TS 風構文・**swc でパース**・API は GAS に寄せた**同期スタイル**（完全互換は謳わない）。
- 実行は **wasmtime 上の QuickJS**（javy 方式・二層隔離）を専用非特権プロセス `script-runtime` で。
  ms 級起動・実行ごと使い捨て・fuel/メモリ/wall-clock 制限・有界実行（長時間 await 不可）。
- 外界は `Shiki.*` ホスト関数ブリッジのみ（能力ゲートウェイで通常認可・監査）。**npm import は v1 不可**。
- 3つの顔: ①ワークフローの script ノード ②スプレッドシートのカスタム関数/マクロ ③skill 実装。
  script から名前指定でワークフロー起動可（起動 API・script 自体は durable にならない）。

### FR-14 skill＆スキルストア
> 設計正本: [miniapp-platform.md](./miniapp-platform.md) §4。skill artifact の構成（SKILL.md相当の指示文＋
> 知識スコープ＋許可ツール＋モデル既定＋few-shot＋任意script）は FR-7 が正。本FRは配布・呼び出し面。

- skill = FR-7 で定義するアーティファクト（同一 kind）。
- 呼び出し面: **①チャット開始時の初期コンテキスト適用（FR-7）②エージェント（ツールとしてマウント）
  ③ワークフロー（skill ノード）**。②③の実効権限は常に宣言スコープ ∩ 実行主体 ReBAC。
- **スキルストア = FR-11 レジストリの再利用**（不変 publish・信頼ティア・同意インストール・署名バンドル）。
  外部 SaaS 連携（Slack 通知等）は first-party skill として少数公式提供。

### FR-15 シークレット管理
> 設計正本: [miniapp-platform.md](./miniapp-platform.md) §5 / design §4.11

- write-only/use-only（平文読み返し API なし・実行時注入のみ・自動レダクト）。
- ReBAC オブジェクト（owner/can_use・同意フローで付与）。**宛先束縛**（宣言ホスト以外への添付を fail-closed 拒否）。
- envelope encryption（`KeyProvider` トレイト: Cloud KMS / ローカルキーファイル）。利用イベントを毎回監査。

### FR-16 フィードバック＆ヘルプ
> 設計正本: design §4.12

- **フィードバックは二段同意**: 既定=メタデータのみ（評価＋カテゴリ＋trace_id・本文なし）／内容の共有
  （本文添付・**自由記述コメント含む**＝顧客コンテンツ扱い）=ユーザー明示同意＋組織管理者ポリシーで禁止可能
  （禁止時はカテゴリ選択のみ）。テナント境界を跨ぐ唯一のデータフローとして監査。
- **ヘルプ**: `help/` 1ソース → ヘルプセンター UI＋RAG 組み込み知識スコープ「shiki-help」（全テナント同梱）。
  チャットから「できること」を検索・質問できる。👎カテゴリをヘルプの穴検出器に使う。

### FR-17 開発者向けドキュメント・法令対応
- 開発者向け: ミニアプリ SDK/CLI・公開 API・skill 開発ガイド（FR-11 の SDK 配布と同一生成物から）。
- 法令: 利用規約・プラポリ・特商法表記は弁護士マター（チェックリストを docs に保持）。
  エンジニアリング対応は**テナント消去機構・データレジデンシ・監査エクスポート**（FR-9）に集約。
  ISMS/SOC2 は認証取得の別トラック（アルファ後・営業上早期着手）。

## 4. 非機能要件（NFR）

| ID | 区分 | 要件 |
|----|------|------|
| NFR-1 | セキュリティ | permission-aware の厳密性（二段authz）。サンドボックスは**ティア別に隔離保証を明示**（wasm=プロセス＋wasm二層〔ブラウザ級〕／gVisor=ユーザー空間カーネル／Firecracker=VM級・KVM前提）。「VM級」を無条件に謳わない。egressデフォルト遮断。生成UIはアンビエント権限なし。 |
| NFR-2 | エアギャップ | オンプレ版は外部API無しで**コア機能**成立可能（LLM/埋め込み/reranker/OCRをローカルGPUで自己ホスト）。**「外部接続必須機能」を明示区分**（web検索・地図タイル・LiteLLM 経由外部モデル等）し、エアギャップでは機能単位で無効化（コアは無傷）。 |
| NFR-3 | 可搬性 | クラウド/オンプレを4〜5本のトレイト差し替えで吸収。アプリ本体は同一。 |
| NFR-4 | 配布容易性 | オンプレは docker compose 一発起動（小規模）／k8s（本番）。部品点数を最小化。 |
| NFR-5 | 日本語 | 埋め込み・全文検索（形態素）・OCR・資料を日本語前提で高品質に。 |
| NFR-6 | 監査性 | 全操作・認可判定・RAG引用を記録し、LLMトレースと相関。 |
| NFR-7 | 拡張性 | モジュラモノリス＋トレイト境界。必要部分のみ後でサービス切り出し可能。 |
| NFR-8 | SaaS（優先） | 全データアクセスを認可コンテキスト経由に統一し、**tenant_id を day-1 から保持**してマルチテナント隔離を成立させる（共有コントロールプレーン＋顧客ごと隔離 cell・design §4.1.1）。 |
| NFR-9 | バックアップ/DR | テナント単位バックアップ・復元手順・RPO/RTO を宣言。Postgres・blob・Qdrant・FGA は**整合スナップショット**（別々に復元して権限とデータがズレる事故を防ぐ・design §4.12）。 |
| NFR-10 | 消去・レジデンシ | 解約/削除要求に対する**テナント完全消去＋バックアップからの期限付き消滅証明**。データは東京リージョン固定、外部 LLM の国外処理はモデルカタログで明示。 |
| NFR-11 | プロビジョニング自動化 | 契約→cell プロビジョニング（OpenTofu・CI 経由）→初期管理者招待までを 1 操作で自動化（手作業プロビジョニングを残さない）。 |

## 5. デプロイ部品（オンプレ）

- アプリ: `shiki-server`（Rustモジュラモノリス）／`sandbox-orchestrator`（特権・別プロセス）／
  `ingestion-worker`（Python・Docling）。
- ステートフル依存: Postgres、Qdrant、OpenFGA、Keycloak、MinIO、（監視）Grafana/Tempo/Loki/Prometheus、Langfuse。
- ローカル推論（エアギャップ時）: vLLM（生成LLM）、埋め込み（Ruri等）、reranker、OCR。
- **受注ごとにHW構成（特にGPUサイジング）を検討**: 同時実行数 × モデルサイズ × コンテキスト長 → 必要VRAM。
