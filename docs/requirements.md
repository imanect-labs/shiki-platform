# shiki 要件定義書

> 本書は設計インタビュー（grill-me）で合意した内容を要件として確定したもの。
> 関連文書: [設計書](./design.md) / [ROADMAP](./roadmap.md)

## 1. プロダクト概要

shiki は **LLMチャットとストレージを核とするエンタープライズ向けAIプラットフォーム**。
権限を厳密に守った文書検索（permission-aware RAG）、サンドボックス上で自律動作するAIエージェント、
社内ツールが自然増殖する generative UI を提供する。SaaS版（マネージド）とオンプレ版の両方を、
**単一のコードベース**から提供する（SaaS を優先ターゲットとする）。

### 1.1 提供形態

| 形態 | 配布 | テナンシー | 推論 |
|------|------|-----------|------|
| **SaaS版（優先）** | Google Cloud（マネージド） | マルチテナント：**共有コントロールプレーン（Keycloak/Org/請求）＋顧客ごと隔離データプレーン（cell）**（design §4.1.1） | マネージド/外部API or ローカル |
| **オンプレ版** | k8s、小規模は docker compose | シングルテナント（1社専有） | 完全ローカル成立可（エアギャップ対応） |

- **SaaS版を最優先で開発**する。オンプレ版は「同一成果物を顧客が自社ホストする」位置づけ（4〜5本のトレイト差し替えで吸収）。
- **単一コードベース原則は維持**：SaaS/オンプレ差はトレイト裏とデプロイ構成で吸収し、アプリ本体は分岐させない。
- マルチテナントを最初から前提にするため、**認可コンテキストの `tenant_id` の継ぎ目を day-1 から確保**する（後付けで隔離境界を壊さない）。

## 2. 前提・基本方針

- **開発の進め方**: 機能ごとにレイヤー横断（API→サービス→ストレージ→フロント）で縦スライス実装する。
- **モダン構成**: サーバ=Rust、フロント=Next.js/TypeScript、モノレポ。
- **コアは自作**: サンドボックス制御層、RAGパイプライン、agent-core、llm-gateway、FUSEを自作する。
- **枯れた技術に乗る領域**: 隔離プリミティブ（Firecracker/gVisor）、認可エンジン（OpenFGA/SpiceDB）、
  認証（Keycloak）、文書パース（Docling）は自作しない（正しさ・運用がクリティカルなため）。
- **可搬性は4〜5本のトレイトに集約**: `ObjectStore` / `VectorStore` / `LlmProvider` / `Sandbox` /
  `DocumentParser` の実装差し替えでクラウド/オンプレを吸収。アプリ本体は無変更。

## 3. 機能要件（FR）

### FR-1 認証・認可基盤
- **認証(AuthN)**: Keycloak。OIDC/SAML/LDAP/AD フェデレーション（顧客IdP連携）＋ローカルIdPの両対応。
- **認可(AuthZ)**: ReBAC（OpenFGA または SpiceDB）。認可の真実を authz store に一元化。
- **組織階層**: 部署などの階層をテナント内に持たせ、柔軟な権限管理。テナント=会社、部署=テナント内ReBAC構造。
- **共有**: ストレージのファイル/フォルダ共有、チャットスレッドの共有を ReBAC relation で表現
  （閲覧/コメント/編集など）。
- **skillex統合 / マルチサービス境界**: 詳細は FR-1.1。

### FR-1.1 マルチサービス境界（shiki × skillex）
- **統一は SaaS 版限定**。オンプレ版は shiki・skillex とも**認証基盤を切り離して単独運用**
  （ローカルKeycloak＋顧客AD、外部依存ゼロ）。エアギャップ要件と矛盾させない。
- **SaaS版の二層**: マルチテナントな**共有コントロールプレーン**（Keycloak＋Org/Member＋請求＋管理ダッシュボード）の下に、
  顧客ごと隔離の**データプレーン**（shikiセル、skillexバックエンド）がぶら下がる。
- **3層の統一/分離境界**:
  1. **User（個人ID）= 統一**。1人1アカウント、共有Keycloakのアイデンティティ。
  2. **Org・メンバー招待・サービスアクセス権（＋サービス管理者の粗い区分）= 統一**（共有プレーンが保持）。
     「サービスロール付与」は `そのサービスを使えるか＋サービス管理者か` 程度の粗い粒度のみ。
  3. **サービス内の細かいロール/ReBAC・各サービスの管理/設定 = 分離**（各データプレーンが独立に保持）。
     shikiの部署階層・ReBAC（OpenFGA）は shiki 内に閉じる。skillexは自前の認可。
- **管理画面はUIのみ統一（データは分離）**: SaaS版は1つの統一「アカウント管理画面」（統一シェル＋共有ページ）に、
  各サービスの設定ページをマイクロフロントエンドで合成し、ページが分かれているだけの一体UIにする。
  ただし各ページは自分のサービスのAPI/ストアを叩き、**authz・設定データは分離のまま**。
  各サービスの管理ページは「シェル埋め込み／単独」両対応の自己完結モジュールとし、オンプレでは単独の管理画面として動く。
- **請求（統一）× 利用量（分離）**: 各サービスが利用量を自前計測（分離）し、**集約使用量のみ**共有請求プレーンへ送る。
  請求書は **Org単位で1枚・サービス別内訳**。プラン/サブスクはサービス別、束ねて1請求。
  **クォータ/上限の強制は各サービス側**、共有プレーンは金額集約のみ。
- **所有**: 共有コントロールプレーンは **shiki repo 所有の SaaS専用モジュール**。skillex は消費側
  （OIDCログイン・サービスアクセス権参照・利用量送信・DLC/LLMトークン取得）。
  shiki-server の AuthN 向き先は**設定で差し替え**（SaaS=共有issuer / オンプレ=ローカルKeycloak）。
- **契約（コントラクト）の正本は shiki repo**: skillex は**別リポジトリ**。skillex が参照する契約
  （OIDC設定、サービスアクセス権API、利用量集約イベントのスキーマ、DLC/LLMトークンの aud/scope）は
  **shiki repo の `contracts/` に正本を置き公開**する。skillex 側はこれを取り込んで実装し、齟齬を防ぐ
  （スキーマ変更は shiki repo 側のバージョン管理・後方互換ポリシで管理）。

### FR-2 ストレージ
- Google Drive 風の基本ファイル操作（アップロード/ダウンロード/フォルダ階層/移動/共有）。
- **3つの顔**を持つ: ①Drive風UI操作、②RAGインデックス対象、③AIエージェントがマウントするファイルシステム。
- **全アクセスは単一 `StorageService` 経由**（権限チェック＋監査＋イベント発行のチョークポイント）。
  バケット直アクセス禁止。
- **バージョニング＋コンテンツアドレッシング**（重複排除、破壊的編集の安全網）を最初から。
- 書込時に**イベント発行**し、RAG増分再索引をトリガ。

### FR-3 permission-aware RAG（精度の良い高速な文書検索）
- **閲覧不可ファイルは検索結果にもLLM回答にも絶対に混入しない**（厳密な権限境界）。
- **二段authzフィルタ**: ①ベクタ/全文の検索時に可読タグで pre-filter、②取得後に authz store で post-filter 検証。
- **ハイブリッド検索**: Dense（埋め込み）＋ Keyword（BM25・日本語形態素）を RRF で融合 → reranker で並べ替え。
- **日本語重視・差し替え可能な埋め込み**（デフォルト Ruri、`embedding_model_version` をベクタに刻む）。
- **高品質インジェスト**: PDF/docx/pptx/xlsx をレイアウト・表・読み順を保って構造化、スキャンPDFは日本語OCR。
- **監査**: 「どの文書のどのチャンクを引用して回答したか」をチャンク単位で記録。

### FR-4 LLMチャット
- スレッド（会話）→ メッセージ。メッセージ内容は**構造化ブロック配列**
  （text / tool_call / tool_result / citation / generative_ui / file_ref）。
- **ツール実行**: コードインタプリタ、文書検索、（クラウド版は）Web検索など。
  - **ユーザーが選択していなくてもLLMが自動判断してツールを使用**（デフォルト全提示・モデル自動選択、
    権限/破壊/コスト系のみ明示許可）。
- **引用ソースの表示**、**SSEストリーミング**（トークン・ツールイベント・生成UIを逐次配信）。
- スレッド共有（ReBAC）。初版は線形会話、ただしブランチ可能な構造。

### FR-5 サンドボックス＆AIエージェント
- **サンドボックス＝隔離された汎用実行環境**（プリミティブ）。その上に用途が乗る：
  - **自律エージェント**: Claude Code/Codex 級。①コード/計算の実行、②ファイルの作成/編集/削除、
    ③任意ツール・任意コマンドの実行。永続ワークスペース＋FUSEでストレージをマウント。
  - **コードインタプリタ/単発実行**: まっさら・短命・実行して破棄、ネット遮断などで強く制約。
- **単一 `agent-core`** がチャットのツール実行と自律エージェントの双方を駆動（差はツールセット/自律度/環境のみ）。
- **egress デフォルト遮断＋allowlist**（機密データ持ち出し防止、エアギャップ対応）。

### FR-6 generative UI ＆ 社内ミニアプリ
- チャット内でAI生成UIコンポーネントをレンダリング。
- **段階戦略**: まず宣言的コンポーネント・カタログ（任意コード実行なし・安全）。
  これは二層モデルの **A（宣言的・in-trust）**。コードベースの **B** と業務アプリ基盤は **FR-11** で扱う。
- **生成UIにアンビエント権限を与えない**。データアクセスは必ず宣言済み・認可済みバックエンドアクション経由。
- **ミニアプリ = prompt template ＋ UIスペック ＋ 許可ツール**のバージョン付きアーティファクト、ReBACで部署共有。

### FR-7 prompt template（gem/GPTs相当）
- 構成: ①システムプロンプト ②知識スコープ（参照を許すフォルダ/タグ＝RAG範囲限定）
  ③許可ツール ④モデル/パラメータ既定 ⑤（任意）few-shot。
- 知識スコープで参照範囲を絞れるが、**最終可読性は常に個人のReBACで再チェック**。
- 部署単位で共有（ReBAC）、バージョン管理。

### FR-8 資料作成
- パワポ/ワード/エクセル生成。
- **v1**: ライブラリ直接生成（xlsx=`rust_xlsxwriter`、docx/pptx=Python worker）。
  既存ひな型のプレースホルダ穴埋め方式を併設。Euro-Office/LibreOffice は変換用途に限定。
- **v2（将来）**: ブラウザ内編集（OnlyOffice Docs / Collabora Online を組込、StorageServiceへ保存→RAG再索引）。
  自作は最終手段。

### FR-9 エンタープライズ機能
- アカウント管理、管理画面、**監査ログ**（操作・権限判定・引用チャンクまで）。

### FR-10 監視基盤
- **インフラ可観測性**: OpenTelemetry ＋ Grafana スタック（Tempo/Loki/Prometheus）。
- **LLM可視化**: Langfuse（プロンプト/補完/トークン/コスト/ツール/レイテンシ/引用チャンク）。
- **監査ログと Langfuse トレースを trace_id で突合**（「誰が・どの権限で・どのチャンクを引用した回答を得たか」）。

### FR-11 ミニアプリ／業務アプリ基盤
FR-6 の generative UI を土台に、「社内業務アプリが自然増殖する」基盤へ拡張する。

- **二層モデル**（同一の「アーティファクト＋バージョン＋ReBAC＋監査」共通枠に乗る。違いはランタイムと認可の入口だけ）:
  - **A. 宣言的ミニアプリ（=FR-6）**: LLM/利用者が作る検証済みUIスペック。in-trust・構造的に安全・宣言的バックエンド束縛のみ。
  - **B. コードベース・ミニアプリ（新規）**: shikiチーム/パワーユーザーが任意フロント（＋任意サーバ）コードで実装。
    out-of-trust として隔離実行し、shiki機能は**公開API＋スコープ付きトークン**経由でのみ呼ぶ。
    **B1＝フロントのみ（本命）／B2＝サーバ側ロジックあり（昇格オプション）**。
- **汎用PaaS/DBaaSは作らない**: アプリが要る「データ置き場／実行場所／shikiへの口」を、
  ①管理された構造化データサービス ②既存サンドボックス基盤の再利用 ③公開APIゲートウェイ、の3点で満たす。
- **セキュアな内部API呼び出し（最重要）**:
  - 内部APIは直接公開しない。**公開APIゲートウェイ（BFF）が唯一の入口**で、キュレーション済み・バージョン付き・スコープ付きの能力面だけを再公開。
  - 認可は**ユーザー委譲OAuth2（authcode＋PKCE）**。既存Keycloakを認可サーバに再利用（新規認証基盤なし）。
  - **二重ゲート**: 実効権限 = アプリ付与スコープ ∩ 呼出ユーザーのReBAC。
    アプリが広いスコープを持っても本人が閲覧不可なデータには到達不可（confused deputy 対策）。
  - ユーザー不在の自動化のみ、**所有データに限定した**狭スコープのサービスidentityを許可。全呼び出しを監査。
- **能力カタログ＆スコープ**: storage / data / rag / ai / identity / events。
  `<能力>.<操作>` ＋ リソース束縛の2軸。スコープは上限、実認可は常にOpenFGA。アプリは自分の所有リソースを持てる。
- **ミニアプリ内AI**: raw `llm.invoke` ＋ `agent.invoke`（ツール＝アプリ宣言 ∩ template ∩ ユーザーReBAC）。
  コンテキストにアプリデータ/RAGを供給可（RAGは個人ReBAC再チェック）。コストは(ユーザー×アプリ)で計上、モデル/予算ガードレール、既存監査に統合。
- **構造化データサービス**: 既存Postgres上の管理サービス（`record` JSONB ＋ スキーマレジストリ ＋ 宣言フィールドへの式インデックス、ランタイムDDLなし）。
  フィールド型に user/dept/file/record 参照を含む。
  **行レベル認可 = テーブルReBAC（OpenFGA）＋クエリ時述語（ABAC・WHERE強制付与・集計にも適用）＋フィールドマスク＋個別共有のみスパースtuple**。
  宣言的クエリ/保存ビュー（生SQL非公開）。レコードのリビジョン履歴。
- **ワークフロー/プロセス管理**: テーブルに紐づく軽量FSM（自作・artifact）。状態=statusフィールド、遷移認可=上記述語の再利用、
  statusが可視性を駆動、副作用は宣言的アクション（AI含む）、サーバ強制＋監査、条件分岐/並列承認まで。重いBPMNエンジンは入れない。
- **ランタイム隔離**: B1=別オリジン＋CSP（connect-srcはゲートウェイ限定・ホスト無権限）／
  B2=既存サンドボックス（Firecracker/gVisor）＋egressデフォルト遮断+allowlist。
- **配布・デプロイ**: ミニアプリ=マニフェスト付きアーティファクト（要求スコープ/所有テーブル/ワークフロー/許可モデル/バンドル/エントリ）。
  内部レジストリに不変publish→管理者が同意してインストール（所有テーブル自動プロビジョン＋ReBAC付与）。
  信頼ティア（first-party署名＝既定信頼／in-house＝管理者同意／将来marketplace＝審査）。
  オンプレ/エアギャップは署名バンドルでネット不要インポート。SDK＋CLI（`shiki app init/dev/publish`）で「簡単にデプロイ」。

## 4. 非機能要件（NFR）

| ID | 区分 | 要件 |
|----|------|------|
| NFR-1 | セキュリティ | permission-aware の厳密性（二段authz）。サンドボックスはVM級隔離。egressデフォルト遮断。生成UIはアンビエント権限なし。 |
| NFR-2 | エアギャップ | オンプレ版は外部API無しで全機能成立可能（LLM/埋め込み/reranker/OCRをローカルGPUで自己ホスト）。 |
| NFR-3 | 可搬性 | クラウド/オンプレを4〜5本のトレイト差し替えで吸収。アプリ本体は同一。 |
| NFR-4 | 配布容易性 | オンプレは docker compose 一発起動（小規模）／k8s（本番）。部品点数を最小化。 |
| NFR-5 | 日本語 | 埋め込み・全文検索（形態素）・OCR・資料を日本語前提で高品質に。 |
| NFR-6 | 監査性 | 全操作・認可判定・RAG引用を記録し、LLMトレースと相関。 |
| NFR-7 | 拡張性 | モジュラモノリス＋トレイト境界。必要部分のみ後でサービス切り出し可能。 |
| NFR-8 | SaaS（優先） | 全データアクセスを認可コンテキスト経由に統一し、**tenant_id を day-1 から保持**してマルチテナント隔離を成立させる（共有コントロールプレーン＋顧客ごと隔離 cell・design §4.1.1）。 |

## 5. デプロイ部品（オンプレ）

- アプリ: `shiki-server`（Rustモジュラモノリス）／`sandbox-orchestrator`（特権・別プロセス）／
  `ingestion-worker`（Python・Docling）。
- ステートフル依存: Postgres、Qdrant、OpenFGA、Keycloak、MinIO、（監視）Grafana/Tempo/Loki/Prometheus、Langfuse。
- ローカル推論（エアギャップ時）: vLLM（生成LLM）、埋め込み（Ruri等）、reranker、OCR。
- **受注ごとにHW構成（特にGPUサイジング）を検討**: 同時実行数 × モデルサイズ × コンテキスト長 → 必要VRAM。

- 会話のブランチUI（構造は先に用意）。
