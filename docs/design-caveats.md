# 設計上の落とし穴・要注意点（実装前に必ず詰める）

> 本書は設計レビューで露見した「このまま実装すると壊れる／詰まる／主張が嘘になる」箇所を、
> 実装着手前に**意思決定して潰すための注意リスト**として固定する。
> 正本は [設計書](./design.md) / [要件](./requirements.md) / [ROADMAP](./roadmap.md)。本書は設計を変えるものではなく、
> 設計が**暗黙にしている前提を明示し、各 Phase の受け入れ条件に反映させる**ためのもの。
>
> 各項目: **箇所 → リスク → 実装前に決めること → 守る不変条件/受け入れ条件**。
> 重大度 🔴 Critical（前提が崩れると当該 Phase が成立しない）/ 🟠 Major / 🟡 Minor。

## 🔴 PIT-1: `authz_tags` 前段フィルタの正体を確定する（permission-aware RAG の心臓）

- **箇所**: design.md §4.3 / phase-2 Task 2.2・2.4・2.7。
- **リスク**: ReBAC の可読性は派生的（直接 viewer / 祖先フォルダ / 部署メンバ / 上位部署…の論理和）。
  これをベクタ検索の集合積フィルタに落とす方式が**未定義**。看板機能の中核アルゴリズムが「authz_tags 対応」の4文字で省略されている。
- **実装前に決めること（二択を明示せよ）**:
  - **(a) ACL展開方式**: chunk に「読める subject 全列挙」を焼く → 部署共有でタグ爆発、再共有で大量再書込。
  - **(b) 権限定義オブジェクト方式（推奨）**: chunk に `folder_id / dept_id` 等の**権限定義オブジェクト**を持たせ、
    検索時に**ユーザーの可読オブジェクト集合を OpenFGA `ListObjects` で算出** → `tag IN (集合)` で絞る。
  - (b) を採るなら **`ListObjects` のカーディナリティ非有界問題**に必ず対処する:
    可読集合を folder/dept 粒度に抑え、**集合が閾値超なら pre-filter を諦め post-filter 全依存に切替える**フォールバックを設計に書く。
- **守る/受け入れ条件**: §4.3 に「採用方式・`ListObjects` 上限・タグ再評価のコストモデル」を明記するまで Task 2.4/2.7 を着手しない。
  「最もベスト」を名乗る前提条件＝本項の解決。

## 🔴 PIT-2: post-filter は top-k を壊す（recall が静かに溶ける）

- **箇所**: phase-2 Task 2.6→2.7 の順序（retrieve top-k → RRF → reranker → OpenFGA post-filter）。
- **リスク**: pre-filter が不完全だと rerank 後に大半が deny され、最終引用件数が非決定的に激減。
  reranker が post-filter の**前**にあり、読めない chunk に計算を浪費。CLAUDE.md「全件取得→フィルタ禁止」とも衝突。
- **実装前に決めること**: **over-fetch 係数＋不足時バックフィル**ループを 2.7 仕様に入れる。
  可能なら認可フィルタを **reranker の前**へ寄せ（pre-filter を信頼できる file 粒度で確定）、post は例外検証のみにする。
- **受け入れ条件**: 「最終引用件数が要求 top-k を下回らない（バックフィルが働く）」を 2.7 に追加。

## 🔴 PIT-3: grant 方向の遅延を検査する（「混入ゼロ」の裏面）

- **箇所**: phase-2 Task 2.7（テストが deny 方向＝権限剥奪のみ）。
- **リスク**: pre-filter タグは非同期更新。**権限付与直後**はタグが追いつかず、対象文書が**エラーも出さず検索に出ない**（under-recall）。
  deny は post-filter が救うが、grant は誰も救わない。「共有したのに出てこない」は典型クレーム。
- **実装前に決めること**: grant 後の**可視化 SLA**（付与→N秒以内に検索に出る）を定義。
- **受け入れ条件**: 2.7 に「共有付与→N秒以内に当該 chunk が検索に現れる」adversarial テストを追加（deny と対で）。

## 🔴 PIT-4: FUSE × StorageService の認可/監査は syscall 粒度では破綻する

- **箇所**: design.md §4.2「FUSE の read/write は裏で StorageService を叩き権限/監査/再索引を必ず通る」/ §4.6。
- **リスク**: FUSE は `stat/open/read(128KB単位)/readdir` を1操作で大量発行。各 syscall に OpenFGA check ＋ 監査 insert を乗せると
  サンドボックス内の `ls`/`grep` が実用不能な遅さになり、監査ログが syscall で溢れる。
  **これは sync/async（§4.2「sync 妥協可」）の問題ではなく、認可・監査の粒度が syscall 単位であること**が原因。
- **実装前に決めること**: マウント時に「このワークスペース部分木への **capability**」を1回発行し、
  以降の syscall は capability 検証のみ（毎回 OpenFGA を叩かない）。監査は syscall ではなく**論理操作（open した node）粒度**で記録。
- **守る**: §4.2/§4.6 に capability モデルを追記。「必ず通る」の意味を「capability 発行時に1回通す」と再定義。

## 🔴 PIT-5: エージェントの read-after-write 一貫性が無い（Phase 5 が動かない）

- **箇所**: design.md §4.2 + phase-2 Task 2.8（再索引「数秒〜分」）+ phase-5。
- **リスク**: エージェントは FUSE 上でファイルを編集しつつ `doc_search` する。書込→**非同期**再索引のため、
  **自分が今書いた内容が索引に未反映で stale を読む**。書込速度 ≫ GPU 索引速度で**キューが永久に追いつかない**シナリオすらある。
  ローカル書込と全体索引の一貫性モデルが存在しない。
- **実装前に決めること**: エージェントのワークスペースを「索引対象」と切り離す。
  エージェントの `doc_search` 系は**ワークスペース直読み（grep/file_read ツール）を別経路**にし、
  ストレージRAG索引は**コミット相当の明示操作時のみ**更新する。
- **守る**: phase-5 設計に「ワークスペース直読み経路」と「索引反映の明示境界」を明記。

## 🟠 PIT-6: presigned URL は単一チョークポイントを破る

- **箇所**: phase-1 Task 1.2「presigned URL は StorageService の権限判定を経た発行のみ」。
- **リスク**: presigned URL は**発行後 MinIO を直接叩き、有効期間中 StorageService を素通り**。
  「バケット直アクセス禁止」「必ず StorageService 経由」「全アクセス監査」(NFR-6) と矛盾。剥奪してもURLは生き続け、実バイト取得は無監査。
- **実装前に決めること（どちらかに倒す）**:
  - (a) 全バイトを StorageService がプロキシ（チョークポイント厳守・コスト増）。
  - (b) presigned を許すが **極短TTL＋発行を監査＋剥奪時の失効手段**を設計に明記。
- **守る**: 「両方OK」を排し、Task 1.2 にどちらを採るか書く。

## 🟠 PIT-7: `doc_chunk` を OpenFGA オブジェクトにするとタプル爆発

- **箇所**: design.md §4.1 ReBAC 図の `doc_chunk -> inherits -> file`。
- **リスク**: chunk 単位を authz store のオブジェクトにすると百万オーダのタプルが乗り、PIT-1 の `ListObjects` をさらに悪化。
  post-filter check は **file 粒度で十分**で、chunk タプルは純オーバーヘッド。
- **実装前に決めること**: 認可の最小オブジェクトは **file**。chunk→file 対応は RAG 側メタで持ち、OpenFGA に chunk を入れない。
- **守る**: §4.1 の図から `doc_chunk` 関係を削除（または「論理的継承であり tuple 化しない」と注記）。

## 🟠 PIT-8: 埋め込みモデル更新が「全停止イベント」になる

- **箇所**: design.md §4.3 / phase-2 Task 2.3「version 変更＝全再構築」かつ「混在を検出・拒否」。
- **リスク**: エアギャップ・百万文書で全再埋め込みはローカルGPUで数日。検索品質が崩れる。
  「混在拒否」ガードが**ローリング移行（新旧並走）を禁止**し、ゼロダウンタイム移行の道を自ら塞ぐ。
- **実装前に決めること**: **shadow index への背面再構築＋エイリアス切替**を前提化。
  「混在拒否」ではなく「**インデックス単位で version 固定・検索はエイリアス先**」に変更。
- **守る**: §4.3 に shadow→切替の移行手順を追記。Task 2.3 の「混在拒否」をインデックス単位の制約に緩める。

## 🟠 PIT-9: llm-gateway の内部正規形＝OpenAI互換は核に対して逆効果

- **箇所**: design.md §4.5 / phase-3 Task 3.2「内部正規形＝OpenAI互換、薄いアダプタ」。
- **リスク**: 本製品の核はエージェント（agent-core）で、agentic 用途は **Claude（Opus/Sonnet）主力が自然**。
  OpenAI スキーマ正規形は Anthropic の **tool_use/tool_result ブロック・トップレベル system・extended thinking・
  prompt caching ブレークポイント・citations** を綺麗に乗せられず、アダプタは「薄く」ならない。最良モデルの機能を最小公倍数で削る。
- **実装前に決めること**: 内部正規形を**プロバイダ中立な content-block 表現**（tool_use/thinking/citation を一級市民に）にし、
  OpenAI/Gemini をアダプト側にする。最低でも prompt caching と thinking を正規形に持つ。デフォルトモデルは Claude 前提で書く。
- **守る**: §4.5 の「OpenAI互換正規形」を「中立 content-block 正規形」に改める。

## 🟠 PIT-10: Phase 3「最初のデモ」が最大リスク（正しさクリティカルな二段authz）に直結

- **箇所**: roadmap phase-2.md（2.10 が 2.7 必須、phase-3 の 3.4 が 2.10 必須）。roadmap.md の「簡易パース先行」前倒し案が phase-2 DoD に反映されていない。
- **リスク**: 最初の顧客価値が、研究的に一番危ない 2.6/2.7（正しさクリティカル）でゲートされる。
- **実装前に決めること**: phase-2 に**二段DoD**を正式化する。
  - **Tier-1**: file 粒度の単純権限フィルタ（OpenFGA file check のみ・chunk タグ無し）で 2.10/Phase 3 を通す。
  - **Tier-2**: authz_tags による高速 pre-filter（PIT-1）は後追い最適化。
- **守る**: 初デモを RAG 高速化研究から切り離す（Tier-1 で M2 到達可能にする）。

# data 行レベル ABAC（§4.10 / Phase 9 Task 9.3-9.4）

> 「テーブルの行ごとに誰が見えるか」の4階層（①テーブルReBAC ②行ポリシー述語 ③フィールドマスク ④個別共有）は
> 方針として筋がいい。だが「見せない」の実装が甘く、**集計・並べ替え・テーブル間参照という"隙間"から中身が漏れる**。

## 🔴 PIT-17: 集計（件数/合計/平均）から、見えない個人がバレる

- **箇所**: Task 9.3「述語は集計にも適用」/ 9.4 aggregate。
- **リスク**: 行フィルタを正しく効かせても、**集計結果の数値そのもの**から個人が漏れる。
  「役職=役員 かつ 部署=X の人数=1」→ 個人特定。条件を少し変えた**2回の合計の差分**→ 特定個人の値を逆算（推論攻撃・スモールセル開示）。
  「行を見せる/見せない」とは別問題で、「集計にも述語適用」では塞げない。「RAGと同一保証」を謳うなら対処が要る。
- **決めること**: 集計をどこまで自由に許すか。**少人数セルの丸め/拒否（最小件数しきい値）**、同一ユーザーの反復差分集計の抑制を仕様化。

## 🔴 PIT-18: 行述語の"材料"が別システム（OpenFGA）にあり、量が膨らむ

- **箇所**: Task 9.3 `$user.dept(subtree)` ＋ 個別共有 `OR id IN (共有id)`。
- **リスク**: 「自部署とその配下の行だけ」の述語には**配下部署ID集合**が要るが、それは組織図＝OpenFGA にあり、行データは Postgres。
  1クエリごとに OpenFGA を複数回叩いて集合を組み立て SQL に貼る構造になる。権限の広い人は配下部署が数千→`IN(数千)` が
  膨らみ遅い/組み立て不能。**PIT-1（RAGの可読集合）と同型の `ListObjects` カーディナリティ問題**。
- **決めること**: 集合の上限・キャッシュ・閾値超のフォールバックを PIT-1 と統一方針で設計。述語の材料取得コストを §4.10 に明記。

## 🔴 PIT-19: フィールドマスクが「取得後マスク」だと、並べ替え/絞り込み/集計で漏れる

- **箇所**: Task 9.4 field_policy「取得後マスク」。
- **リスク**: 給与列を表示マスクしても、**その列で sort/filter/aggregate ができれば中身は推測できる**
  （高い順ソート＝序列が見える、`給与>1000万` で絞れば答え、平均は PIT-17 に合流）。「表示を隠す」と「検索に使わせない」は別物。
- **決めること**: マスク対象フィールドは **filter/sort/aggregate の対象からも除外**する、と 9.4 に明記。

## 🔴 PIT-20: lookup/record参照のテーブル間で権限が素通りする

- **箇所**: Task 9.2 フィールド型 lookup/record参照・計算 / 9.3（単一テーブルの述語のみ記述）。
- **リスク**: 見えないテーブルAの値を、見えるテーブルBの **lookup列経由で釣れる**（データ層の confused-deputy）。
  9.3 は1テーブルの述語しか定義せず、**参照先テーブルの行ポリシーの再帰適用**が抜けている。kintone代替に参照は必須機能。
- **決めること**: lookup/参照/計算は**参照先テーブルの行ポリシーも透過適用**する、と 9.3 に明記（結合時の述語伝播）。

## 🟠 PIT-21: 「WHERE強制注入」は必要条件にすぎない

- **箇所**: Task 9.3「省略不可・上書き不可のWHERE述語をAND付与・生SQL非公開」。
- **リスク**: 入口を塞ぐ方針は正しいが十分ではない。Postgres RLS ですらエラーメッセージ・型変換失敗のタイミング・
  関数副作用から漏れた実例がある。PIT-17〜20 はその漏れ口の具体例。基盤全体で最も正しさがクリティカルな領域を1行で済ませている。
- **決めること**: 漏れ口（集計・マスク・参照・エラー差分）の脅威モデルを列挙し、各々に対策テストを 9.3 受け入れ条件へ。

# サンドボックス（§4.6 / Phase 4）

> 隔離プリミティブ（Firecracker/gVisor）は既製で正しい。落とし穴は**自作する制御層**にある。

## 🔴 PIT-22: 「<200ms 起動」と「ユーザーごとの束縛」が衝突する

- **箇所**: Task 4.5 温機プール<200ms / 4.6 egress allowlist / 4.9 FUSE マウント / 9.12 secret 注入。
- **リスク**: 温機プールは**誰が使うか決まる前に**事前起動する。だが FUSE マウント（そのユーザーのストレージ）・
  egress allowlist・B2 の secret 注入は**払い出し時に初めて確定**する。この束縛時間が 200ms 予算の中に入る。
  設計は「高速起動」と「ユーザー隔離」を独立に書いているが、**両者は時間予算を奪い合う**。
- **決めること**: 「事前ウォーム済み（ユーザー非依存部分）」と「払い出し時バインド（FUSE/egress/secret）」を分離し、
  後者を含めた end-to-end のレイテンシ目標を 4.5 に書く。スナップショットに secret を焼かない順序を明記。

## 🔴 PIT-23: 「ゲスト→特権オーケストレータ」RPC がサンドボックス脱出の主経路

- **箇所**: Task 4.7 ホスト↔VM ツールRPC（vsock）/ 4.2 特権プロセス。
- **リスク**: orchestrator は**特権**（KVM/jailer/ネットワーク名前空間）。その RPC は**サンドボックス内＝信頼できない側**から
  入力を受ける。orchestrator がゲスト供給データ（パス・サイズ・ファイル名・コマンド結果）を不用意に解釈すると、
  **サンドボックスから特権プロセスへ脱出**＝ホスト侵害。これが古典的かつ最悪の脱出ベクタだが、設計に脅威モデルの記述が無い。
- **決めること**: 「ゲスト入力は全て敵対的」を前提に、RPC の入力検証・パストラバーサル・サイズ/数量制限・
  orchestrator の権限最小化を 4.2/4.7 の受け入れ条件へ。fuzzing を CI に。

## 🟠 PIT-24: gVisor フォールバックは「VM級隔離（NFR-1）」を黙って下げる

- **箇所**: Task 4.4 gVisor（KVM無し環境）/ NFR-1「サンドボックスはVM級隔離」。
- **リスク**: Firecracker は VM級だが gVisor はユーザ空間カーネルで**隔離強度が一段弱く**、脱出 CVE の実績もある。
  KVM が使えないデプロイ（一部マネージドクラウド等）では NFR-1 が実質**未達**になるのに、設計は gVisor を透過的等価として扱う。
- **決めること**: gVisor 時は隔離強度が下がることを**能力フラグで明示**し、機密度の高いワークロード（B2/自律エージェント）は
  gVisor を**拒否 or 警告**するポリシを書く。NFR-1 を「KVM 前提」と条件付きで正直に記す。

## 🟠 PIT-25: egress allowlist を「ホスト名」で守るには SNI 対応プロキシが要る

- **箇所**: Task 4.6「allowlist（宛先ホスト/ポート/CIDR）」。
- **リスク**: ホスト名 allowlist は IP/CIDR では守れない（CDN で IP は変動・広域、DNS 解決自体が egress）。
  外部API（例 Anthropic）を許すには **TLS SNI を見る egress プロキシ**が必要で、IP フィルタとは別物。「ホスト/CIDR」混在は
  機構が未決の兆候。
- **決めること**: egress 制御の機構を1つに決める（SNI 対応 forward proxy 推奨）。allowlist の単位（ホスト名 vs CIDR）を統一。

# skillex マルチサービス境界（§4.1.1 / Phase 0 Task 0.10）

> 「User=統一・サービスアクセス権=統一・細かい認可/データ=分離」の3層は整理として妥当。
> だが**統一した部分が全顧客・両サービスの共有 blast radius**になり、分離の利点を一部打ち消す。

## 🔴 PIT-26: 共有コントロールプレーンが「顧客ごと隔離セル」を identity で破る

- **箇所**: §4.1.1 共有 Keycloak＋Org/請求（マルチテナント）。
- **リスク**: データプレーンは顧客ごと隔離セルなのに、**Keycloak（全顧客の identity）と請求は共有・マルチテナント**。
  共有 Keycloak を1つ陥とすと**両サービスの全顧客**に波及。セル隔離は data plane だけで、identity は共有爆心地。
- **決めること**: 共有プレーンの脅威モデル・テナント分離（realm/issuer 分割の是非）・侵害時の影響範囲を文書化。
  「セル隔離」の主張に identity 共有の例外を明記。

## 🔴 PIT-27: トークンの aud/scope 厳密束縛とサービスアクセス権の参照点

- **箇所**: §4.1.1「サービスアクセス権参照」/ Task 0.10 aud/scope / contracts。
- **リスク**: 共有 identity ゆえ、**skillex 向けトークンが shiki に提示**され得る。aud を厳密検証しないとサービス跨ぎの
  confused-deputy。また「このユーザーが shiki を使えるか」の判定を毎回共有プレーンに問えば **SPOF＋レイテンシ**、
  キャッシュすれば**剥奪が効かない（stale）**。設計はこの参照点と整合性を詰めていない。
- **決めること**: aud/scope の厳密束縛を必須化。サービスアクセス権の**強制点（各サービス側）と失効伝播**（イベント or 短TTL）を contracts に明記。

## 🟠 PIT-28: 「請求=統一×利用量=分離」は金額クリティカルなのに整合保証が無い

- **箇所**: §4.1.1 各サービスが利用量計測→集約のみ請求プレーンへ。
- **リスク**: 利用量イベントは**課金の根拠＝金額クリティカル**。at-least-once で送ると**重複請求/欠損**、改竄・突合の手段が無いと
  過少/過大請求でも検知できない。設計は「集約値のみ送る」とだけ書き、冪等性・突合・監査を欠く。
- **決めること**: 利用量イベントに冪等キー・期間締め・突合（reconciliation）・改竄検知を設計。請求は分離計測の**事後照合**前提にする。

## 🟠 PIT-29: 「設定差し替えだけでオンプレ化」は実態より強い主張

- **箇所**: §4.1.1「AuthN 向き先を設定で差し替え」/ オンプレは共有プレーン非搭載。
- **リスク**: オンプレは issuer を差し替えるだけでなく、**Org管理・サービスアクセス権・請求・管理ダッシュボードという共有プレーンの
  機能群が丸ごと存在しない**。これらに依存するコードはオンプレで**代替実装 or 無効化**が要り、「本体を分岐させない」不変条件と緊張する。
- **決めること**: 共有プレーン依存機能ごとに**オンプレ縮退仕様**（ローカル等価 or 無効）を定義し、本体分岐を避けるトレイト/設定境界を明記。

# ブラウザの認証トークン保管方式

> 「フロントは OIDC JWT を取得し Authorization ヘッダで送信」（design.md §4.1）という現行方式の再検討。
> 詳細な比較・影響範囲・移行段階は [docs/auth/browser-token-strategy.md](./auth/browser-token-strategy.md) を正本とする。

## 🔴 PIT-30: ブラウザの localStorage に JWT を置くと XSS 流出＋失効不能の二重弱点になる

- **箇所**: design.md §4.1「フロントは OIDC JWT を取得し Authorization ヘッダで送信」/ `web/src/lib/auth.ts`（`WebStorageStateStore(localStorage)` + `automaticSilentRenew`）/ `web/src/lib/api.ts`（Bearer 付与）。
- **リスク**:
  - **(1) XSS でトークン総取り**: `localStorage` は任意の JS から可読。1回の XSS で **access + refresh の両方が流出 → 長期アカウント乗っ取り**。`draft-ietf-oauth-browser-based-apps` が localStorage 保管を非推奨とし BFF を推奨する理由そのもの。「権限考慮」を売りにする製品が、セキュリティレビューで確実に指摘される構成を採っている。
  - **(2) 失効が即時に効かない**: JWT は `exp` まで有効。PIT-27（剥奪が stale）と Task 1.6「共有解除で即時アクセス不可」という**製品要件と根本的に相性が悪い**。
  - 当初の JWT 採用根拠（エアギャップ NFR-2 / 隔離セル / フェデレーション / サービス間 identity）は精査すると**ほぼ全てが「内部・サービス間」の話**で、**「ブラウザが JWT を持つこと」を正当化していない**。エアギャップは「外部依存ゼロ」であって「状態ゼロ」ではなく、各セルがローカルなセッションストアを持てば成立する。
- **決めること（二択を明示せよ）**:
  - **(a) BFF + オパークセッション Cookie（推奨）**: ブラウザは `httpOnly`+`Secure`+`SameSite` の不透明セッション Cookie のみ保持。OIDC token はサーバ側に保管し、downstream/サービス間へは JWT/token-exchange で転送。**XSS 流出を遮断＋失効を即時化**。ステートフル化は**ブラウザ⇄api 境界のみ**で、内部/サービス間は JWT でステートレスのまま。SSE は Cookie 自動添付で `EventSource` がそのまま使える（現行の fetch-stream ヘッダ注入が不要に）。
  - **(b) 中間案 — JWT を httpOnly Cookie 化**: localStorage 流出だけ潰す軽量移行。ただし**失効の弱点は残る**（exp まで有効）＋ CSRF 対策が必要。Task 1.6 の即時失効要件を本気で満たすなら結局 (a) まで行く。
  - いずれも **CSRF 対策**（`SameSite` ＋ CSRF トークン / double-submit）を必須化する。
- **Phase 9 の app-gateway(BFF) との区別**: phase-9 Task 9.6 の「BFF」は**ミニアプリ用の公開APIゲートウェイ**（`crates/app-gateway`）であり、本項の**メインWebアプリ認証 BFF**（`crates/api` の auth セッション境界）とは**別物**。混同しないこと。
- **決定（2026-06-18 human 合意）**: **(a) フル BFF + オパークセッション Cookie** を採用。セッションストアは **Redis**、`SameSite=Lax` ＋ double-submit CSRF トークン、**web/api は同一オリジン配信**を前提。design.md §4.1 は更新済み。詳細は [docs/auth/browser-token-strategy.md §7](./auth/browser-token-strategy.md)。
- **守る/受け入れ条件**: 認可・公開API境界の不変条件（confused-deputy 防御・authz バイパス禁止）を維持。実装時に「XSS でトークンが盗めない（ブラウザにトークンを置かない）」「権限剥奪が即時に反映される（セッション削除で失効）」をテストで担保。オンプレ縮退仕様（PIT-29）に Redis 依存を追記すること。
